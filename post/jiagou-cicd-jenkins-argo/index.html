Argo CD 知识点汇总（GitOps 持续交付最佳实践）

适用版本：Argo CD 2.x（当前主流版本） 阅读对象： Kubernetes 运维工程师、DevOps 工程师、平台架构师、CI/CD 工程师

一、什么是 Argo CD

1、Argo CD 概述

Argo CD 是一个运行在 Kubernetes 集群中的 声明式（Declarative）GitOps 持续交付（Continuous Delivery，CD）工具。

它最大的特点就是：

Git 仓库就是整个 Kubernetes 集群的唯一真实来源（Single Source of Truth）。

所有 Kubernetes YAML、Helm Chart、Kustomize 配置都保存在 Git 中。

Argo CD 会不断比较：

 1Git Repository
 2        │
 3        │ Desired State（期望状态）
 4        ▼
 5    Argo CD
 6        │ Compare
 7        ▼
 8 Kubernetes Cluster
 9        ▲
10        │ Actual State（实际状态）

如果发现：

1Git
2Deployment replicas = 3
3
4↓
5
6Kubernetes
7Deployment replicas = 5

Argo CD 就会认为：

1OutOfSync（状态不一致）

随后：

  • 自动恢复（Auto Sync）
  • 或等待人工同步（Manual Sync）

这就是 GitOps。

二十二、Argo CD 与 Jenkins 的关系

很多初学者容易误解 Argo CD 会替代 Jenkins。实际上，两者定位不同，最佳实践是CI 与 CD 分离。

对比项 Jenkins Argo CD
定位 CI（持续集成） CD（持续交付）
主要职责 编译、测试、打包、镜像构建 Kubernetes 应用部署
是否构建镜像 ✅ ❌
是否执行单元测试 ✅ ❌
是否管理 GitOps ❌ ✅
是否持续对比集群状态 ❌ ✅
是否自动修复配置漂移 ❌ ✅（Self Heal）

推荐的企业级流水线

 1开发人员
 2    │
 3    ▼
 4Git Push（应用源码）
 5    │
 6    ▼
 7Jenkins Pipeline
 8    │
 9    ├── 编译（Maven/Gradle/npm）
10    ├── 单元测试
11    ├── 构建 Docker 镜像
12    ├── 推送镜像到 Harbor
13    └── 更新 GitOps 仓库中的镜像 Tag
14                │
15                ▼
16          GitOps Repository
17                │
18                ▼
19             Argo CD
20                │
21        自动检测 Git 变更
22                │
23                ▼
24      Kubernetes 集群自动同步

一句话总结： Jenkins 负责"生产交付物（镜像）"，Argo CD 负责"按照 Git 中声明的状态部署交付物"。

二十三、Argo CD 最佳实践

Git 仓库结构建议

1gitops-repo/
2├── apps/
3│   ├── dev/
4│   ├── test/
5│   └── prod/
6├── helm/
7├── kustomize/
8├── projects/
9└── applications/

环境隔离

  • Dev：自动同步，可开启 selfHeal。
  • Test：自动同步，建议结合自动化测试。
  • Prod：建议保留人工审批或受控同步，避免误发布。

与 Jenkins 集成

  • Jenkins 不直接执行 kubectl apply。
  • Jenkins 只更新 GitOps 仓库中的 Helm values.yaml 或 Kustomize 配置（例如镜像 Tag）。
  • Argo CD 监听 Git 仓库并完成部署，实现发布过程可追溯、可回滚。

权限控制

  • 使用 AppProject 限制不同团队可访问的 Git 仓库、命名空间和 Kubernetes 集群。
  • 对生产环境启用基于角色的访问控制（RBAC）和单点登录（SSO）。

高可用部署

  • API Server、Repo Server、Application Controller 可配置多副本。
  • Redis 建议采用高可用方案。
  • 为关键组件配置持久化存储、资源限制及监控告警（如 Prometheus + Grafana）。

总结

Argo CD 是 Kubernetes 生态中最成熟的 GitOps 持续交付平台，其核心理念是以 Git 为唯一可信源，持续比较 Git 中的期望状态与集群中的实际状态，并自动完成同步、漂移修复和回滚。

对于你最近学习的 Jenkins CI/CD，建议把两者结合起来理解：

  • Jenkins = CI（构建、测试、打包、推送镜像）
  • Argo CD = CD（监听 GitOps 仓库、部署到 Kubernetes）
  • Harbor = 镜像仓库
  • Kubernetes = 应用运行平台

四者组合形成目前企业中最常见的云原生交付架构：

 1Developer
 2    │
 3    ▼
 4 Git（源码）
 5    │
 6    ▼
 7 Jenkins（CI）
 8    │
 9    ▼
10 Harbor（镜像仓库）
11    │
12    ▼
13 GitOps Repository（部署配置）
14    │
15    ▼
16 Argo CD（CD）
17    │
18    ▼
19 Kubernetes Cluster

这也是目前 CNCF 云原生生态中应用最广泛的持续交付模式之一，非常值得系统掌握。
