Elasticsearch中字段与数组类型的关系

即使 tags 定义为 keyword：

1"tags": { "type": "keyword" }

👉 你仍然可以直接存数组：

1"tags": ["tag1", "tag2"]

因为在 Elasticsearch 中：

keyword 类型本质上支持“多值字段（multi-value field）”

🚀 一、ES 为什么允许 keyword 存数组？

📌 关键点：ES 字段本质是“多值结构”

在 ES 中，一个字段不是“单值变量”，而是：

👉 一个字段 = 一个 value list（值集合）

📦 举个例子

你写：

1"tags": ["java", "elasticsearch", "backend"]

ES 实际存储是：

1tags = {
2  "java",
3  "elasticsearch",
4  "backend"
5}

👉 它不是数组概念，而是 多值字段

🔍 二、keyword 的真实含义

keyword 的定义是：

📌 不分词 + 精确索引 + 可用于聚合/排序

但它没有规定：

❌ 必须是单值 ✔ 可以是多值

⚙️ 三、底层索引机制（重点）

ES 底层会把数组拆成多个倒排索引项：

1"tags": ["A", "B", "C"]

会被索引成：

1A → doc1
2B → doc1
3C → doc1

👉 所以查询时：

1term: { tags: "A" }

一样能命中

📌 四、对比理解（非常重要）

类型 是否可数组 是否分词 用途
text ✔ ✔ 模糊搜索
keyword ✔ ❌ 精确匹配
integer ✔ ❌ 数值
object ✔ ❌ 结构

⚠️ 五、容易误解的点（重点）

❌ 误区1：keyword = 单值字段

错！

👉 keyword 只是“不可分词字符串”，不是单值限制

❌ 误区2：数组是特殊类型

错！

👉 ES 没有 array 类型

数组 = 普通字段 + 多值

❌ 误区3：mapping 要声明 array

错！

ES 不需要：

1"type": "array" ❌（不存在）

🎯 八、一句话总结

👉 keyword 能存数组的原因：

Elasticsearch 字段本身就是“多值结构”，数组只是 JSON 表达方式，不是类型限制。

在 Elasticsearch 里，date 类型其实是非常灵活但也容易踩坑的一类字段。现在把它讲透：支持哪些日期格式 + 底层机制 + 推荐写法 + 常见坑

🧠 一、结论先说清楚

date 类型支持三大类格式：

✅ 1. 标准日期格式（推荐）

  • ISO 8601（最常用）
12026-01-25
22026-01-25T10:30:00
32026-01-25T10:30:00Z
42026-01-25T10:30:00+08:00

✅ 2. 时间戳（毫秒/秒）

11706188200000   （毫秒）
21706188200      （秒，需配置）

✅ 3. 自定义格式（mapping 定义）

比如：

125/01/2026
22026-01-25 10:30:00

👉 需要在 mapping 中声明 format

⚙️ 二、默认支持的日期格式（重点🔥）

如果你不写 format，默认是：

1strict_date_optional_time||epoch_millis

等价于：

📌 1. ISO 8601（最推荐）

12026-01-25T10:30:00Z
22026-01-25T10:30:00+08:00

📌 2. 日期 + 时间（可选时间）

12026-01-25
22026-01-25T10:30:00

📌 3. 时间戳（毫秒）

11706188200000

🎯 九、一句话总结

👉 Elasticsearch 的 date 类型支持：

  • ISO 8601（标准）
  • 时间戳（毫秒/秒）
  • 自定义格式（format 定义）
  • 相对时间（now-7d）

要分清一个关键点：ES 接收的是“解析后的时间”，不是你输入的“字符串长什么样” 👍，查询的时候可以使用其他时间格式

在 Elasticsearch 中，date 类型本质上是：

👉 内部统一转成 long（毫秒时间戳）存储

所以结论是：

🧠 一、答案先说清楚

✅ 可以用时间戳查询 ISO 8601 / 日期时间字段

但必须满足：

✔ 1. 时间戳是“合法 epoch millis / seconds”

✔ 2. mapping 支持 epoch 格式

✔ 3. 查询时 ES 会自动转换

⚙️ 二、底层原理（非常重要🔥）

无论你写的是：

12026-01-25T10:00:00Z

或：

12026-01-25 10:00:00

ES 内部都会转成：

1→ 1706176800000 (毫秒时间戳)

👉 所以查询时：

ES 做的是 数值范围比较，不是字符串比较

🚀 如果系统是后端服务（PHP/Java）

👉 建议：

场景 推荐
前端传参 时间戳
ES 查询 转 ISO 8601
ES 存储 ISO 8601

完美解决了 PHP 开发者的痛点：既利用了 ES 强大的时间处理能力，又避免了复杂的字符串时区转换

字段类型设为 date，但在 Mapping 中指定 format 为 epoch_second——完美解决了 PHP 开发者的痛点：既利用了 ES 强大的时间处理能力，又避免了复杂的字符串时区转换。

  1. ES 层面：
    • ES 的 date 类型本质上就是为了存储长整型数字（毫秒或秒）。
    • 当你指定 format: "epoch_second" 后，ES 就会明确知道：“哦，用户给我的这个整数（比如 1715412345），是秒级的，我需要把它存进去，并且在查询时把它当作时间处理。”
    • Kibana/Grafana 支持吗？ 完美支持。只要字段类型是 date，无论底层存的是字符串还是数字，可视化工具都能识别出时间轴，画图、聚合（如按天统计）完全不受影响。
  2. PHP 层面：
    • 写入：直接用 time() 返回整数，简单粗暴，无需 DateTime 格式化。
    • 读取：取出来是整数，直接用 date('Y-m-d H:i:s', $timestamp) 就能转回本地时间，符合 PHP 程序员直觉。

🛠️ 具体实施步骤

1. 创建索引 Mapping（关键一步）

你必须显式定义 Mapping，不能依赖 ES 的动态映射（因为动态映射默认可能优先识别为 epoch_millis 或者字符串）。

 1PUT /my_php_logs
 2{
 3  "mappings": {
 4    "properties": {
 5      "create_time": {
 6        "type": "date",
 7        "format": "epoch_second" 
 8      },
 9      "message": {
10        "type": "text"
11      }
12    }
13  }
14}

2. PHP 写入数据

1$doc = [
2    'create_time' => time(), // 直接传入 10 位整数，例如 1715412345
3    'message'     => '用户登录成功'
4];
5// 存入 ES...

3. PHP 读取数据

1// 假设 $hit['_source']['create_time'] 取出来是 1715412345
2$timestamp = $hit['_source']['create_time'];
3
4// 直接转为可读时间 (注意：PHP 的 date() 默认是服务器时区，通常是东八区)
5echo date('Y-m-d H:i:s', $timestamp);

⚠️ 必须注意的 3 个“坑”

虽然这个方案很好，但有几个细节如果不注意，会导致数据错乱：

1. 秒级 vs 毫秒级（精度问题）

  • 你的方案：使用 epoch_second，精度只有秒。
  • 潜在问题：如果你的业务需要毫秒级精度（比如金融交易、高频日志），time() 是不够的。
  • 解决：如果你需要毫秒，Mapping 要改为 format: "epoch_millis"，PHP 写入时要用 microtime(true) * 1000 或者 intval(microtime(true) * 1000)。

2. 混合格式的陷阱（非常重要！）

ES 的 format 字段支持多种格式组合，用 || 分隔。 ES 解析数字的逻辑是“按顺序匹配”。

  • 场景：如果你定义 format: "epoch_millis||epoch_second"。

  • 坑

    ：当你传入一个 10 位的时间戳（如

    11715412345

    ），ES 会先尝试把它当作

    1epoch_millis

    （毫秒）。

    • 结果：ES 会认为这是 1970 年 1 月 20 日左右的时间（因为数字太小了）。

  • 正确做法

    ：如果你只存秒级，就只写

    1epoch_second

    。如果你既要兼容毫秒又要兼容秒，必须把

    秒级放在前面

    （但这通常不推荐，容易混淆）。

    • 推荐：统一使用一种精度。

3. Kibana 的显示时区

  • 存入 ES 的是 UTC 时间戳（数字）。
  • Kibana 读取后，会根据浏览器的设置自动转换为本地时间显示。
  • 注意：你在 Kibana 看到的时间是转换后的，而你在 PHP 代码里 var_dump 出来的原始数据是 UTC 的。这有时候会造成“时间对不上”的错觉，其实只是显示层面的转换，数据本身没问题。

PHP中查询的时候需要特别注意

如果你仅仅在 Mapping 里写死了 format: "epoch_second"，当你尝试传入 ISO 8601 格式（例如 "2026-05-11T15:43:55+08:00"）进行查询时，Elasticsearch 会直接报错（failed to parse date field），因为它只被允许接收纯数字。

但是，Elasticsearch 的 format 字段非常灵活，支持“多选”逻辑。

为了既保留 PHP 写入时的便利性（直接传 time() 整数），又能在查询时享受 ISO 8601 的直观（直接传字符串），你需要修改 Mapping 配置。

以下是具体的解决方案：

1. 核心方案：修改 Mapping 支持多种格式

你需要使用双竖线 || 来分隔不同的格式。这告诉 ES：“这个字段既可以是秒级时间戳，也可以是 ISO 8601 字符串”。

修改后的 Mapping：

 1PUT /my_php_logs
 2{
 3  "mappings": {
 4    "properties": {
 5      "create_time": {
 6        "type": "date",
 7        "format": "epoch_second||strict_date_optional_time"
 8      }
 9    }
10  }
11}
  • epoch_second：允许你继续用 time() 写入数据。
  • strict_date_optional_time：这是 ES 内置的格式，完美支持 ISO 8601（如 2026-05-11T15:43:55+08:00 或 2026-05-11T07:43:55Z）。

2. 实际效果演示

配置好上述 Mapping 后，你的 PHP 代码可以这样“混用”：

场景 A：写入数据（使用整数，保持你的便利习惯）

1// PHP 代码
2$data = [
3    'id' => 101,
4    'create_time' => time() // 传入整数，例如 1715412345
5];
6// $client->index(...) 
7// 结果：ES 成功识别为秒级时间戳并存储

场景 B：查询数据（使用 ISO 8601，利用时区优势）

 1// PHP 代码
 2// 假设你要查询 2026-05-11 12:00:00 之后的数据
 3$queryTime = '2026-05-11T12:00:00+08:00'; // 明确指定东八区
 4
 5$params = [
 6    'index' => 'my_php_logs',
 7    'body' => [
 8        'query' => [
 9            'range' => [
10                'create_time' => [
11                    'gte' => $queryTime // 传入字符串
12                    // 注意：这里不需要写 'format' => '...' 
13                    // 因为 ES 会根据 Mapping 自动尝试匹配
14                ]
15            ]
16        ]
17    ]
18];
19// $client->search(...)
20// 结果：ES 自动识别这是 ISO 格式，转为 UTC 后与内部存储的时间戳比对，查询成功！

3. 为什么要这样做？（优势分析）

这种“Mapping 组合拳”方案完美结合了你的需求和最佳实践：

维度 你的需求 解决方案带来的好处
写入性能 想要快，不想转字符串 继续用 time() 传整数，PHP 零开销。
查询灵活性 想要准，不想算时区 查询时用 ISO 8601 字符串（带 +08:00），ES 帮你自动处理时区转换，彻底解决“8小时偏差”问题。
可读性 调试方便 在 Kibana 里直接复制粘贴时间字符串就能查，不用去在线工具转时间戳。

⚠️ 重要提示

  1. 顺序很重要：在 format 中，建议把 epoch_second 放在前面（如上面的例子）。虽然 ES 会尝试按顺序匹配，但如果你的时间戳数字比较小（比如早期的时间），放在后面可能会被误判。不过对于现代时间（10位数字），ES 通常能智能区分数字和字符串。
  2. 不要混用精度：如果你定义了 epoch_second||epoch_millis，查询时传入 1715412345（10位），ES 可能会优先匹配第一个（秒级），这是符合预期的。但如果你的 Mapping 是 epoch_millis||epoch_second，传入 10 位数字可能会被误认为是毫秒（导致时间回到 1970 年）。既然你决定用 time()，就务必把 epoch_second 放在格式列表的第一位。

请更新你的 Mapping 为 "format": "epoch_second||strict_date_optional_time"。这样，写入用整数，查询用字符串，两全其美！

📌 总结

对于 PHP 开发者来说，date 类型 + epoch_second 格式 是性价比极高的选择：

  1. 代码极简：不用处理 DateTime 和 Z 后缀。
  2. 性能优秀：数字传输比长字符串快。
  3. 功能完整：Kibana 聚合、Grafana 画图全部支持。

只要记住一点：在 Mapping 里写死 format: "epoch_second"，千万别让 ES 猜。

⚠️ 七、为什么不建议直接用时间戳查？

虽然可以，但有 3 个问题：

❌ 1. 可读性差

11706176800000 ❌

❌ 2. 时区风险

ISO 自动带时区，timestamp 不会

❌ 3. 调试困难

Kibana 里不直观

🎯 八、一句话总结

👉 可以用时间戳查询 ISO 8601 日期字段，因为 ES 内部最终都会转换成毫秒时间戳进行比较

在 Elasticsearch 中乐观锁并发控制

Elasticsearch 的乐观锁本质

就是：

更新时附带“我认为当前文档应该是什么版本”的条件。

如果条件成立：

1允许更新

否则：

1version_conflict_engine_exception

早期版本（6.x、7.x 初期）使用

  • version
  • version_type

因为version 本身也是乐观锁版本号，逻辑版本号

例如：

操作 version
初始 1
修改一次 2
修改一次 3

所以：

1?version=3

就表示：

1我认为当前版本应该是3

但是从 7.x 后期到 8.x，官方已经不推荐甚至限制直接使用 version 做并发控制，而改为：

  • if_seq_no

  • if_primary_term

这也是 Elasticsearch 8.x 官方推荐的 OCC（Optimistic Concurrency Control）机制。

一、为什么旧版(早期版本（6.x、7.x 初期）) version 被淘汰

旧版本的并发控制：

1{
2  "version": 3
3}

的问题：

  • version 是逻辑版本号
  • 主从复制情况下可能不够严谨
  • failover 后 version 可能不准确
  • 无法精确识别主分片切换

因此 Elasticsearch 后来引入：

新版 OCC 机制

基于：

字段 作用
_seq_no 文档操作序列号
_primary_term 主分片任期编号

两者一起才能唯一确定：

“这个文档是不是我刚刚读取的那个版本”

这是 Elasticsearch 现在真正底层的并发控制机制。

二、旧版 version 并发控制（老方式）

1、老版本流程

第一步：查询文档

1GET student/_doc/1

返回：

1{
2  "_index": "student",
3  "_id": "1",
4  "_version": 3,
5  "_source": {
6    "name": "张三",
7    "age": 18
8  }
9}

这里：

1"_version": 3

代表当前版本。

2、更新时携带 version

1PUT student/_doc/1?version=3
2{
3  "name": "张三",
4  "age": 20
5}

3、成功

因为当前版本确实是 3。

更新后：

1"_version": 4

4、并发冲突

如果别人已经更新过：

当前版本已经变成：

1"_version": 4

你再提交：

1PUT student/_doc/1?version=3

就会报错：

1{
2  "error": {
3    "type": "version_conflict_engine_exception"
4  },
5  "status": 409
6}

三、新版本 OCC（推荐方式）

Elasticsearch 8.x 推荐：

if_seq_no + if_primary_term

核心思想

更新时必须满足：

1seq_no 相同
2并且
3primary_term 相同

否则认为：

文档已经被别人修改过。

四、_seq_no 和 _primary_term 在哪里？

  1. 其实默认 GET 是有的。

例如：

1GET student/_doc/1

返回：

 1{
 2  "_index": "student",
 3  "_id": "1",
 4  "_version": 7,
 5  "_seq_no": 15,
 6  "_primary_term": 1,
 7  "found": true,
 8  "_source": {
 9    "name": "张三",
10    "age": 18
11  }
12}

重点：

1"_seq_no": 15,
2"_primary_term": 1

这两个字段就是 OCC 的核心。

  1. Search API 要显式开启

    如果你是：

    1GET student/_search

    你必须加：

    1seq_no_primary_term=true

    例如：

    1GET student/_search?seq_no_primary_term=true

    这时才会返回：

     1{
 2  "hits": {
 3    "hits": [
 4      {
 5        "_id": "1",
 6        "_seq_no": 15,
 7        "_primary_term": 1,
 8        "_source": {
 9          "name": "张三"
10        }
11      }
12    ]
13  }
14}
    为什么 search 默认不返回 seq_no

    因为： seq_no 是内部并发元数据，大量搜索时返回会增加开销。

    所以默认关闭。

    只有：

    1?seq_no_primary_term=true

    才会返回。

五、新版完整并发控制流程

1、读取文档

1GET student/_doc/1

得到：

1{
2  "_seq_no": 15,
3  "_primary_term": 1,
4  "_source": {
5    "name": "张三",
6    "age": 18
7  }
8}

记住：

字段 值
seq_no 15
primary_term 1

2、更新时带上 OCC 条件

1PUT student/_doc/1?if_seq_no=15&if_primary_term=1
2{
3  "name": "张三",
4  "age": 20
5}

3、更新成功

返回：

1{
2  "_seq_no": 16,
3  "_primary_term": 1,
4  "result": "updated"
5}

注意：

seq_no 自动增加。

4、Update API 也是一样

例如：

1POST /student/_update/1?if_seq_no=15&if_primary_term=1
2{
3  "doc": {
4    "age": 20
5  }
6}

同样会启用 OCC。

5、Bulk API 也支持 OCC

例如：

 1{ 
 2  "update": {
 3    "_id": "1",
 4    "_index": "student",
 5    "if_seq_no": 15,
 6    "if_primary_term": 1
 7  }
 8}
 9{ 
10  "doc": {
11    "age": 20
12  }
13}

Bulk 内部也是一样。

6、并发冲突

如果别人已经修改：

当前：

1"_seq_no": 16

你仍提交：

1PUT student/_doc/1?if_seq_no=15&if_primary_term=1

就会报：

1{
2  "error": {
3    "type": "version_conflict_engine_exception",
4    "reason": "[1]: version conflict, required seqNo [15], primary term [1]. current document has seqNo [16] and primary term [1]"
5  },
6  "status": 409
7}

这就是：

乐观锁失败

六、为什么 primary_term 也必须参与

这是很多人忽略的重点。

场景

假设：

  • 节点A 是 primary
  • 节点A 挂了
  • 节点B 晋升为新的 primary

这时候：

1primary_term 会 +1

例如：

时间 primary_term
原主分片 1
切换后 2

这样 Elasticsearch 可以避免：

老 primary 的脏数据覆盖新 primary。

这是 version 做不到的。所以必须：seq_no + primary_term

七、官方推荐的正确姿势

现在 Elasticsearch 官方建议：

读取 → 携带 seq_no + primary_term 更新

标准流程：

1GET doc
2↓
3获取 seq_no/primary_term
4↓
5PUT/UPDATE 时携带
6↓
7成功 or 冲突

这里非常关键 🔥

只要携带：

参数 是否启用 OCC
if_seq_no ✅
if_primary_term ✅

就会启用乐观锁。

如果不带呢？

例如：

1PUT /student/_doc/1

没有：

1if_seq_no
2if_primary_term

那么：

最后一次写入覆盖前一次

也就是：

Last Write Wins

没有并发保护。

下面给你一个企业里真正常见的写法。

查询文档

2    'index' => 'student',

4]);
5
6print_r($response);

返回：

带 OCC 更新

 1$response = $client->update([
 2    'index' => 'student',

九、企业级推荐封装方式

实际开发里通常这样封装：

 1public function safeUpdate(
 2    string $index,
 3    string $id,
 4    array $doc
 5)
 6{
 7    // 1. 查询
 8    $old = $this->client->get([
 9        'index' => $index,
10        'id'    => $id
11    ]);
12
13    // 2. 获取 OCC 信息
14    $seqNo = $old['_seq_no'];
15    $primaryTerm = $old['_primary_term'];
16
17    // 3. 更新
18    return $this->client->update([
19        'index' => $index,
20        'id'    => $id,
21
22        'if_seq_no' => $seqNo,
23        'if_primary_term' => $primaryTerm,
24
25        'body' => [
26            'doc' => $doc
27        ]
28    ]);
29}

十、retry_on_conflict 是什么？

很多人会看到：

1'retry_on_conflict' => 3

例如：

 1$client->update([
 2    'index' => 'student',
 3    'id'    => '1',
 4
 5    'retry_on_conflict' => 3,
 6
 7    'body' => [
 8        'doc' => [
 9            'score' => 99
10        ]
11    ]
12]);

它表示：

如果 OCC 冲突，自动重试 3 次。

适合：

  • 计数器
  • 点赞
  • 浏览量

不适合：

  • 财务
  • 库存
  • 关键业务

因为可能覆盖别人数据。

十一、两种方式对比

对比项 version（旧） seq_no + primary_term（新）
ES8 是否推荐 ❌ ✅
是否准确 一般 很高
是否支持主分片切换 ❌ ✅
是否适合分布式 一般 非常适合
是否官方推荐 ❌ ✅
企业是否使用 很少 标准方案

_version 还存在

注意：

ES8 并没有删除：

1"_version"

你仍然能看到。

但是：

不再推荐用于 OCC 并发控制

现在：

字段 用途
_version 内部版本信息
_seq_no 真正并发控制
_primary_term 主分片任期控制

十二、重点结论（非常重要）🔥

1、ES8 以后不要再使用 version 做 OCC

应改为：

1if_seq_no
2if_primary_term

2、_seq_no 和 _primary_term 是系统字段

查询文档时自动返回：

1"_seq_no"
2"_primary_term"

3、真正企业级流程

标准做法：

1先 GET
2↓
3获取 seq_no/primary_term
4↓
5再 UPDATE

4、retry_on_conflict 不等于真正安全

它只是：

自动重试。

不是严格乐观锁。

5、Update API 本质也是 OCC

实际上：

1update
2=
3get + reindex

底层仍然使用 seq_no 机制。

一句话总结：

老版本

1version
2=
3逻辑版本乐观锁

新版本

1if_seq_no + if_primary_term
2=
3分布式安全乐观锁

两者共同本质

都是：

1只有“当前版本”符合预期
2才允许更新

否则：

1409 conflict

因为：

version 不够“分布式”

而：

1seq_no + primary_term

才真正适合：

  • 分片
  • 主从
  • failover
  • replication
  • cluster recovery

这些 ES 核心场景。
