2026-01-28 · 8 分钟阅读 · Elasticsearch  ·
分享到:

在 Elasticsearch 中，如果一个字段需要同时满足全文检索（使用 text）和精确匹配、聚合或排序（使用 keyword）的需求，最佳实践是使用 多字段（Multi-fields）映射。

这种配置允许你在同一个字段名下，同时拥有两种不同的索引方式。

🛠️ Mapping 映射配置示例

你可以参考以下标准的 JSON 格式来定义 Mapping：

 1{
 2  "mappings": {
 3    "properties": {
 4      "goods_name": {
 5        "type": "text",  // 主类型：text，用于全文检索
 6        "fields": {
 7          "keyword": {
 8            "type": "keyword",  // 子字段：keyword，用于精确匹配、聚合、排序
 9            "ignore_above": 256  // 超过256字符的字符串不索引，优化性能
10          }
11        }
12      }
13    }
14  }
15}

💡 如何查询与使用

配置完成后，同一个字段名就可以根据不同的业务场景，通过不同的路径来调用：

  • 全文检索：直接使用主字段名（如 goods_name），配合 match 等查询语句。
    • 例如：match: { "goods_name": "智能手机" }
  • 精确匹配 / 聚合 / 排序：使用带后缀的子字段名（如 goods_name.keyword），配合 term 查询或聚合语句。
    • 例如：term: { "goods_name.keyword": "华为Mate60 Pro" }

⚙️ 核心参数说明

  • type: "text"：该字段的内容会被分词器拆分，建立倒排索引，适合模糊搜索和全文检索。
  • type: "keyword"：该字段的内容会作为整体被索引，不会被分词，适合精确查找、过滤、聚合统计和排序。
  • ignore_above：这是 keyword 类型的一个重要参数。它限制了字符串的最大索引长度（默认通常是 256 个字符）。超过这个长度的字符串将不会被 keyword 子字段索引，这样可以有效防止超长字符串占用过多内存，提升聚合和排序的性能。

在 Elasticsearch 的多字段（Multi-fields）映射中，fields 里面的子字段名只是你为了方便调用而起的一个“别名”或“后缀”。虽然大家习惯将其命名为 keyword，但这并不是强制的硬性规定。

💡 自定义子字段名称示例

你可以根据业务语义或者个人习惯来命名。例如，你可以把 keyword 子字段命名为 raw（原始值）、exact（精确值）或者 agg（专门用于聚合）：

 1{
 2  "mappings": {
 3    "properties": {
 4      "product_name": {
 5        "type": "text",
 6        "fields": {
 7          "raw": {          // 自由定义的子字段名
 8            "type": "keyword",
 9            "ignore_above": 256
10          },
11          "pinyin": {       // 另一个自由定义的子字段名（比如用于拼音搜索）
12            "type": "text",
13            "analyzer": "pinyin_analyzer"
14          }
15        }
16      }
17    }
18  }
19}

📌 调用时的注意事项

自定义名称后，你在查询、聚合或排序时，只需要把后缀换成你定义的名字即可：

  • 如果你定义的是 raw，调用时就使用 product_name.raw。
  • 如果你定义的是 exact，调用时就使用 product_name.exact。

总结一下： 只要符合 Elasticsearch 的字段命名规范，子字段叫什么名字完全由你决定。使用默认的 .keyword 是一种广泛认可的社区最佳实践（别人接手代码时更容易看懂），但在实际开发中，根据你的具体业务场景（比如专门分出 .pinyin 做拼音搜索、.english 做英文分词等）来灵活命名是非常常见且推荐的做法。

在 Elasticsearch 的多字段（Multi-fields）映射中，确实不能无限制地定义子字段。

每一个多字段（包括主字段和它下面的所有子字段）都会被算作一个独立的字段，并受到 Elasticsearch 索引级别的严格限制。

⚙️ 核心限制与默认值

Elasticsearch 为了防止“映射爆炸（Mapping Explosion）”导致内存溢出和集群性能崩溃，设置了以下几个关键的硬性限制：

  1. 字段总数限制（最直接的限制）

    • 参数：index.mapping.total_fields.limit
    • 默认值：1000
    • 说明：这是最核心的限制。一个索引中所有字段的总数（包含主字段、多字段中的子字段、对象映射以及字段别名）不能超过这个数值。如果你为一个主字段定义了 10 个子字段，那么它一共会消耗 11 个字段配额。

  2. 嵌套深度限制

    • 参数：index.mapping.depth.limit
    • 默认值：20
    • 说明：限制字段的嵌套层级。例如，a.b.c 的深度为 3。如果你的多字段或对象嵌套过深，会触发此限制。

  3. 字段名长度限制

    • 参数：index.mapping.field_name_length.limit
    • 默认值：无限制（Long.MAX_VALUE）
    • 说明：限制单个字段名称的字符长度。虽然默认不限，但在极端情况下也可以进行约束。

💡 应对策略与建议

  • 按需定义，避免浪费：不要为了“以防万一”而随意添加大量用不到的子字段。每增加一个子字段，不仅会消耗字段配额，还会增加内存占用和集群状态（Cluster State）的负担。
  • 谨慎调高限制：虽然你可以通过修改索引的 settings 来调高 index.mapping.total_fields.limit 的上限（例如调至 2000），但这通常不建议。过高的字段数量会显著降低 Elasticsearch 的性能，增加内存压力，甚至可能导致集群难以恢复。
  • 使用 Index Templates（索引模板）：如果你希望统一管控所有新建索引的字段上限，可以在索引模板中提前设置好这些限制参数，从而在源头上规避风险。

简单来说，虽然多字段功能非常强大且灵活，但在设计 Mapping 时依然需要保持克制，做好合理的规划。

⚠️ 注意事项

  1. 新索引配置：上述 Mapping 最好在你新建索引（Index）的时候直接定义好。
  2. 已有索引修改：如果索引已经存在，且该字段的类型与你想修改的不一致，Elasticsearch 不支持直接修改已有字段的数据类型。此时，你需要新建一个索引并配置好正确的 Mapping，然后将旧索引的数据重新导入（Reindex）到新索引中。
  3. 新增子字段：如果原字段已经是 text 类型，你可以通过更新 Mapping 的方式为其新增一个 keyword 类型的子字段，这通常是允许的。

多字段中的子字段、对象映射以及字段别名这些概念的区别，这三个概念在 Elasticsearch 的 Mapping 中非常核心，虽然它们都涉及到“字段”，但解决的问题和底层原理完全不同。我们可以用生活中的例子来快速区分：

  • 多字段（子字段）：给同一份数据拍“不同滤镜的照片”（一份用于模糊搜索，一份用于精确匹配）。
  • 对象映射：一个“收纳盒”，里面装着有结构的小物件，但拿出来的时候会被打散平铺。
  • 字段别名：给字段起的一个“外号”，你叫外号时它也会答应，但户口本上只有原名。

以下是它们的具体区别和实例说明：

1. 多字段中的子字段 (Multi-fields)

核心作用：“一源多态”。即同一份原始数据，为了满足不同的查询需求（如全文检索、精确匹配、排序、聚合），建立多套不同的索引方案。

  • 底层原理：原始值只存一份（在 _source 中），但在倒排索引中会根据不同的类型（如 text 和 keyword）建立多份索引。
  • 典型场景：一个商品名称，既需要分词搜索，又需要按完整名称排序。

举例说明：

 1"product_name": {
 2  "type": "text",  // 主字段：分词，用于全文搜索
 3  "fields": {
 4    "keyword": {   // 子字段：不分词，用于精确匹配和排序
 5      "type": "keyword"
 6    },
 7    "pinyin": {    // 子字段：使用拼音分词器
 8      "type": "text",
 9      "analyzer": "pinyin"
10    }
11  }
12}
  • 搜 product_name: "手机"（走 text 索引，模糊匹配）
  • 排序用 product_name.keyword（走 keyword 索引，按完整字符串排序）

2. 对象映射 (Object Mapping)

核心作用：“结构化嵌套”。用于在文档中存储 JSON 对象或嵌套的键值对。

  • 底层原理：Elasticsearch 默认会将对象内部的字段**“扁平化（Flatten）”**。也就是说，对象的层级结构在索引时会被打散，合并到顶层字段中。
  • 典型场景：存储用户的收货地址（包含省、市、街道等固定结构）。

举例说明： 假设你写入这样的 JSON 数据：

1{
2  "user_name": "张三",
3  "address": {
4    "city": "南昌",
5    "street": "红谷滩大道"
6  }
7}

在 Elasticsearch 的底层索引中，address 对象的层级会被打散，实际存储的字段变成了 address.city 和 address.street。你可以直接通过点符号（.）来查询，如 term: { "address.city": "南昌" }。

在 Elasticsearch 的对象映射（Object Mapping）中，处理三级甚至更深度的嵌套 JSON 结构是非常常见的。

为了让你更直观地理解，我们继续用“用户信息”来举例。假设我们要存储一个用户的详细档案，其中包含一个三级的地址结构：address（一级） -> location（二级） -> geo（三级）。

📝 1. 原始 JSON 数据（三级结构）

这是你准备存入 Elasticsearch 的原始业务数据：

 1{
 2  "user_name": "张三",
 3  "address": {
 4    "city": "南昌",
 5    "location": {
 6      "district": "红谷滩区",
 7      "geo": {
 8        "lat": 28.682,
 9        "lon": 115.857
10      }
11    }
12  }
13}

🛠️ 2. 对应的 Mapping 映射定义

在定义 Mapping 时，我们需要像剥洋葱一样，一层一层地把 properties 嵌套下去：

 1{
 2  "mappings": {
 3    "properties": {
 4      "user_name": {
 5        "type": "keyword"
 6      },
 7      "address": {                // 一级对象
 8        "type": "object",
 9        "properties": {
10          "city": {
11            "type": "keyword"
12          },
13          "location": {           // 二级对象
14            "type": "object",
15            "properties": {
16              "district": {
17                "type": "keyword"
18              },
19              "geo": {            // 三级对象
20                "type": "object",
21                "properties": {
22                  "lat": { "type": "float" },
23                  "lon": { "type": "float" }
24                }
25              }
26            }
27          }
28        }
29      }
30    }
31  }
32}

注：在 Elasticsearch 7.x 及以上版本中，type: "object" 其实是默认值。即使你不显式写出 "type": "object"，只要写了 properties，ES 也会自动将其识别为对象映射。

⚙️ 3. 底层存储真相：扁平化处理

这是对象映射最核心的一点。虽然你存入和定义时是三层嵌套的 JSON，但 Elasticsearch 的底层搜索引擎（Lucene）无法理解层级对象。

因此，ES 在内部索引时，会自动将这三级结构**“扁平化（Flatten）”**成简单的键值对，并用点号（.）连接层级。

底层实际存储的字段结构如下：

  • user_name: "张三"
  • address.city: "南昌"
  • address.location.district: "红谷滩区"
  • address.location.geo.lat: 28.682
  • address.location.geo.lon: 115.857

🔍 4. 如何查询三级嵌套字段

正因为底层被扁平化了，你在查询（Query）、过滤（Filter）或聚合（Aggregation）时，只需要使用**点号（.）**将层级串联起来即可，完全不需要使用复杂的嵌套查询语法。

例如，查询住在“红谷滩区”的所有用户：

1{
2  "query": {
3    "term": {
4      "address.location.district": "红谷滩区"
5    }
6  }
7}

⚠️ 5. 对象映射的局限性（重要提醒）

对象映射非常适合存储单个、固定结构的层级数据（比如一个用户只有一个收货地址）。

但如果你的三级结构是一个对象数组，对象映射就会出问题。比如一个用户有多个地址：

1"address": [
2  { "city": "南昌", "type": "home" },
3  { "city": "北京", "type": "work" }
4]

在扁平化后，city 变成了 ["南昌", "北京"]，type 变成了 ["home", "work"]。ES 会丢失“南昌”和“home”原本是一对的关联关系。如果你遇到这种对象数组且需要精确组合查询的场景，就需要使用 nested（嵌套类型） 来替代 object 类型。

3. 字段别名 (Field Alias)

核心作用：“只读快捷方式”。给已经存在的实际字段起一个替代名称，主要用于查询和聚合阶段。

  • 底层原理：它不存储任何数据，也不建立新的索引，仅仅是一个指向实际字段的“指针”或映射关系。
  • 典型场景：字段改名迁移（为了兼容旧代码）、或者让字段名符合某些规范（如 ECS 规范）。
  • 致命限制：字段别名只能用于查询（读），绝对不能用于写入数据。

举例说明：

 1"mappings": {
 2  "properties": {
 3    "actual_price": {      // 实际存储数据的字段
 4      "type": "float"
 5    },
 6    "price_alias": {       // 字段别名
 7      "type": "alias",
 8      "path": "actual_price" // 指向实际字段
 9    }
10  }
11}
  • 查询时：你可以用 GET /index/_search { "query": { "range": { "price_alias": { "gte": 100 } } } }，效果和查 actual_price 一模一样。
  • 写入时：如果你尝试向 price_alias 写入数据，ES 会直接报错（illegal_argument_exception: alias cannot be used in write operations）。

📌 核心区别总结

概念 核心作用 是否额外占用存储/索引 写入数据时
多字段 (子字段) 同一数据，多种索引方式（如分词+不分词） 是（会建立多份倒排索引） 写入主字段，自动应用到子字段
对象映射 存储有层级的 JSON 对象结构 否（字段会被扁平化处理） 直接写入 JSON 对象即可
字段别名 给字段起外号，方便查询 否（只是一个只读指针） 禁止写入，只能查

nested（嵌套类型）是 Elasticsearch 中专门用来解决对象数组查询问题的“神器”。

在上一轮对话中我们提到，普通的 object 类型在处理数组时会把字段“扁平化”，导致对象之间的关联关系丢失。而 nested 类型就是为了解决这个痛点而生的。

🎯 为什么需要 Nested 类型？（痛点解析）

假设你有一个商品文档，里面包含了多个不同规格（颜色、尺寸）的库存对象。如果你使用默认的 object 类型，会发生什么？

1. 存入的数据：

1{
2  "product": "T-Shirt",
3  "variants": [
4    { "color": "red", "size": "M" },
5    { "color": "blue", "size": "L" }
6  ]
7}

2. object 类型的底层扁平化存储： ES 会把它拆散成两个独立的数组：

  • variants.color: ["red", "blue"]
  • variants.size: ["M", "L"]

3. 产生的致命问题： 此时，如果你去查询 “颜色是红色 且 尺寸是 L” 的商品：

 1{
 2  "query": {
 3    "bool": {
 4      "must": [
 5        { "term": { "variants.color": "red" } },
 6        { "term": { "variants.size": "L" } }
 7      ]
 8    }
 9  }
10}

结果会错误地匹配到这件商品！ 因为在扁平化的数组里，ES 发现确实有 red，也确实有 L，但它已经忘了 red 原本是和 M 配对的。这就是“对象数组的关联关系丢失”。

🛠️ Nested 类型的实战范例

使用 nested 类型，ES 会将数组中的每个对象作为独立的隐藏子文档来存储，从而保证查询时字段依然绑定在一起。

1. 定义 Mapping 映射

在定义时，必须显式地将该字段声明为 nested 类型：

 1PUT /products
 2{
 3  "mappings": {
 4    "properties": {
 5      "product": { "type": "text" },
 6      "variants": {
 7        "type": "nested",  // 核心：声明为嵌套类型
 8        "properties": {
 9          "color": { "type": "keyword" },
10          "size": { "type": "keyword" }
11        }
12      }
13    }
14  }
15}

2. 写入数据

写入的数据格式和普通对象数组完全一样，不需要做任何特殊处理：

1POST /products/_doc/1
2{
3  "product": "T-Shirt",
4  "variants": [
5    { "color": "red", "size": "M" },
6    { "color": "blue", "size": "L" }
7  ]
8}

3. 使用 nested 语法查询

查询嵌套对象时，不能直接用普通的 term 或 match 查询，必须使用专门的 nested 查询语法，并指定 path（路径）：

 1GET /products/_search
 2{
 3  "query": {
 4    "nested": {
 5      "path": "variants", // 指定要查询的嵌套字段路径
 6      "query": {
 7        "bool": {
 8          "must": [
 9            { "term": { "variants.color": "red" } },
10            { "term": { "variants.size": "L" } }
11          ]
12        }
13      }
14    }
15  }
16}

结果： 这次不会匹配到任何文档。因为 nested 查询会深入到每一个独立的子文档中去匹配，它会发现 red 对应的尺寸是 M，而 L 对应的颜色是 blue，没有任何一个子文档能同时满足 red 和 L，从而保证了查询的绝对精准。

💡 Nested 类型的底层原理与特性

  • 独立子文档：在底层（Lucene引擎），nested 会把上面的数据存成 1 个主文档 + 2 个隐藏的子文档。每个子文档都独立包含一组完整的 color 和 size 信息。

  • 必须使用专用语法：你不能用普通的 variants.color: red 去查，必须包裹在 nested 查询中。

  • inner_hits（返回匹配的子对象）：默认情况下，查询结果会返回整个主文档（包含所有的 variants）。如果你只想高亮或返回那个真正匹配成功的子对象，可以使用 inner_hits 参数：

    1"nested": {
2  "path": "variants",
3  "query": { ... },
4  "inner_hits": {} // 加上这个，结果里会单独列出匹配的 variants 对象
5}

⚖️ Object vs Nested：如何选择？

特性 Object 类型 (默认) Nested 类型
存储方式 扁平化，数组内字段合并 每个对象存为独立的隐藏子文档
查询准确性 数组对象交叉匹配（易出错） 对象内字段独立精准匹配
查询语法 普通查询 (term, match) 必须用 nested 专用查询
性能开销 低（存储紧凑） 较高（维护子文档，查询需额外计算）
适用场景 单个对象（如用户的唯一地址） 对象数组且需跨字段精准查询（如订单明细、商品SKU）

总结建议： 如果你的业务数据是单个对象（比如一个用户只有一个家庭地址），直接用默认的 object 即可，性能最好。但如果你的数据是对象数组（比如一个订单里有多个商品，或者一篇博客有多条评论），并且你需要同时根据数组内对象的多个属性进行精准筛选（比如“找评论作者是Alice且评分大于4星的订单”），那么 nested 类型是你唯一且必须的选择。

Elasticsearch倒排索引的不可变性是指一旦索引数据被写入磁盘，原始的索引文件永远不会被修改，而是通过创建新的索引段来实现数据更新。

1. 不可变性的核心含义

当文档被索引到Elasticsearch后，其倒排索引数据一旦写入磁盘，原始的索引文件永远不会被直接修改。这意味着：

  • 无法直接更新或删除磁盘上的现有索引数据
  • 所有更新操作（包括新增、修改、删除）都是通过创建新的索引段来实现
  • 旧的索引段保持不变，只会在后台合并过程中被清理

2. 为什么设计为不可变

优势

  • 无需锁机制：因为索引文件不会被修改，所以不需要担心多进程同时修改数据的问题
  • 缓存效率高：索引被读入内核文件系统缓存后会一直保留，只要缓存空间足够，大部分读请求直接从内存获取，避免磁盘I/O
  • 数据压缩：单个大型倒排索引可被压缩，减少30%-70%的磁盘空间占用
  • 查询缓存稳定：filter等缓存无需在数据变化时重建，因为数据不会变化

挑战

  • 无法直接更新现有文档
  • 删除操作只是标记而非物理删除
  • 需要额外机制处理更新和删除

3. 动态更新如何实现

Elasticsearch通过多段索引+合并机制解决不可变性带来的挑战：

3.1 基本工作流程

  1. 内存缓冲：新文档首先写入内存索引缓存(Index Buffer)
  2. 定期刷新：默认每1秒执行一次refresh操作，将内存数据写入新的不可变段(Segment)
  3. 查询合并：查询时会同时搜索所有段，然后合并结果
  4. 后台合并：定期将小段合并成大段，清理已删除文档

3.2 更新与删除的特殊处理

  • 删除操作：旧文档不会立即物理删除，而是在.del文件中被标记为已删除
  • 更新操作：旧版本文档被标记删除，新版本文档被索引到新段中
  • 查询过滤：标记为已删除的文档在最终结果返回前会被自动过滤掉

4. 实际例子说明

假设有一个文档："谷歌地图之父跳槽Facebook"，其倒排索引片段如下：

1Term: 谷歌 → [Doc1;1;<1>]
2Term: 地图 → [Doc1;1;<2>]
3Term: 之父 → [Doc1;1;<3>]
4Term: 跳槽 → [Doc1;1;<4>]
5Term: Facebook → [Doc1;1;<5>]

现在需要更新为"谷歌地图之父加盟Facebook"：

  1. 旧文档被标记为已删除（在.del文件中记录）
  2. 新文档被写入内存缓冲区
  3. Refresh后，新文档被索引到新的段中： 
    1Term: 谷歌 → [Doc1;1;<1>], [Doc1;1;<5>]  // 注意位置变化
2Term: 地图 → [Doc1;1;<2>]
3Term: 之父 → [Doc1;1;<3>]
4Term: 加盟 → [Doc1;1;<4>]
5Term: Facebook → [Doc1;1;<7>]
  4. 查询时：系统会同时查询旧段和新段，但旧段中标记为删除的文档会被过滤掉

5. 段合并的重要性

随着时间推移，会产生大量小段和标记删除的文档，Elasticsearch通过后台合并机制解决：

  • 定期合并小段为大段
  • 物理删除已标记为删除的文档
  • 优化存储：减少文件句柄消耗，提高查询效率

例如，合并前：

1段A: [文档1, 文档2, 文档3]  // 文档3已标记删除
2段B: [文档4, 文档5]

合并后：

1新段C: [文档1, 文档2, 文档4, 文档5]  // 文档3被物理移除

6. 与传统数据库的对比

与B+树索引不同，倒排索引采用空间换时间的设计哲学：

  • B+树索引：直接修改磁盘上的索引结构，需要锁机制
  • 倒排索引：通过不可变段和合并机制，牺牲空间换取查询性能

这种设计使Elasticsearch能在亿级文档规模下仍保持毫秒级响应，特别适合近实时搜索场景。

总结来说，倒排索引的不可变性是Elasticsearch高性能的核心设计之一，通过巧妙的段机制和合并策略，既保证了查询效率，又能实现数据的动态更新。

在 Elasticsearch 中，将 must 和 filter 结合使用是一种非常经典且高效的查询模式，通常被称为“全文检索 + 结构化过滤”。

这种组合的核心思想是：

  • must: 负责全文检索，计算相关性得分（_score），决定结果的排序。
  • filter: 负责精确或范围过滤，不计算相关性得分，但可以利用缓存，性能极高。

下面通过一个电商场景的范例来详细讲解。

🛒 业务场景

假设你需要在一个商品库中进行搜索，需求如下：

  1. 全文搜索：查找商品标题中包含“智能手机”的商品。
  2. 精确过滤：品牌必须是“华为”。
  3. 范围过滤：价格需要在 2000 到 5000 元之间。
  4. 状态过滤：商品必须是已上架状态。

💻 DSL 查询范例

针对以上需求，你可以构建如下的查询语句（DSL）：

 1GET /products/_search
 2{
 3  "query": {
 4    "bool": {
 5      "must": [
 6        {
 7          "match": {
 8            "title": "智能手机"
 9          }
10        }
11      ],
12      "filter": [
13        {
14          "term": {
15            "brand.keyword": "华为"
16          }
17        },
18        {
19          "range": {
20            "price": {
21              "gte": 2000,
22              "lte": 5000
23            }
24          }
25        },
26        {
27          "term": {
28            "status": 1
29          }
30        }
31      ]
32    }
33  }
34}

🤔 为什么这是最优组合？

这种 must + filter 的模式之所以是最佳实践，主要基于以下两点：

1. 职责分离，结果更精准

  • must 子句：使用 match 查询对 title 字段进行全文检索。Elasticsearch 会使用 BM25 等算法分析“智能手机”与商品标题的匹配程度，并为每个匹配的文档计算一个相关性分数（_score）。这确保了最相关的商品会排在结果的前面。
  • filter 子句：用于处理那些“非黑即白”的条件。品牌、价格范围、上架状态都是精确匹配，不存在“相关性”一说。将它们放在 filter 中，可以避免这些条件干扰 _score 的计算，使排序完全由全文检索的匹配度决定。

2. 性能优化，查询更快速

  • 不计算分数：filter 上下文中的查询不计算 _score，这本身就节省了大量的 CPU 计算资源。
  • 结果可缓存：这是 filter 最大的性能优势。Elasticsearch 会自动缓存 filter 的查询结果（通常使用 bitset）。当有大量查询都包含 status: 1 或 brand: 华为 这类条件时，系统可以直接从缓存中获取匹配的文档集合，而无需重复扫描索引，性能提升可达 5-10 倍。

简单来说，你可以将这个过程想象成：filter 先快速地从海量商品中筛选出一个符合所有硬性条件（品牌、价格、状态）的“候选池”，然后 must 再在这个“候选池”里，根据标题的相关性进行打分和排序，最终返回给你最理想的结果。

📝 最佳实践口诀

为了方便记忆，你可以参考这个简单的原则：

  • 全文检索、影响排序的条件 → 放入 must (或 should)
  • 精确匹配、范围过滤、不影响排序的条件 → 放入 filter

在 Elasticsearch 中，bool 查询是最核心、最强大的复合查询方式，它允许你将多个独立的查询子句通过布尔逻辑组合起来，构建出非常复杂的查询条件。

bool 查询主要包含以下四种方式（或称子句）：

🧩 四种核心子句

这四种方式共同构成了 bool 查询的基石，它们分别是 must、filter、should 和 must_not。

must (必须匹配)

  • 逻辑：类似于 SQL 中的 AND。文档必须满足 must 数组中所有的查询条件才会被返回。
  • 算分：会计算相关性得分 (_score)。满足的条件越多、匹配度越高，最终得分也越高。
  • 用途：通常用于全文检索等需要衡量相关性的场景，例如搜索标题或描述中包含特定关键词的文档。

filter (过滤)

  • 逻辑：同样，文档必须满足 filter 数组中所有的查询条件。
  • 算分：不会计算相关性得分，所有匹配的文档得分均为 0。
  • 用途与优势：用于精确匹配（如状态、分类ID）或范围查询（如价格、时间）。最大的优势是性能极高，因为查询结果可以被缓存，非常适合处理大量数据时的硬性过滤。

should (应该匹配)

  • 逻辑：类似于 SQL 中的 OR。文档满足 should 数组中的一个或多个查询条件即可。
  • 算分：会计算相关性得分。满足的 should 条件越多，文档的得分越高。
  • 特殊规则：
    • 如果 bool 查询中没有 must 或 filter 子句，那么 should 子句中至少需要匹配一个条件。
    • 如果 bool 查询中存在 must 或 filter 子句，那么 should 子句是可选的，即使一个都不匹配，只要满足了 must/filter，文档也会被返回。
  • 用途：用于提升相关性，例如在搜索结果中，如果文档同时满足某些额外条件（如标签匹配），则给予加分，使其排名更靠前。

must_not (必须不匹配)

  • 逻辑：类似于 SQL 中的 NOT。文档绝对不能满足 must_not 数组中的任何查询条件。
  • 算分：不会计算相关性得分。
  • 用途：用于排除不希望出现的结果，例如排除已下架的商品、特定品牌的商品等。

⚙️ 关键参数：minimum_should_match

这个参数用于精确控制 should 子句的匹配行为。它定义了 should 子句中至少需要满足的条件数量。

  • 默认值：0。
  • 作用场景：当 bool 查询中存在 must 或 filter 时，should 默认是可选的。通过设置 minimum_should_match，可以强制要求 should 中也必须满足指定数量的条件。
  • 示例："minimum_should_match": 2 表示 should 数组中的条件至少要满足 2 个。也可以使用百分比，如 "50%"。

📊 四种方式对比

为了方便理解和记忆，可以参考下表：

子句 逻辑关系 是否影响 _score 主要用途
must 必须满足 (AND) 是 全文检索，决定相关性排序
filter 必须满足 (AND) 否 精确/范围过滤，高性能，可缓存
should 可选满足 (OR) 是 提升相关性得分，或作为可选条件
must_not 必须不满足 (NOT) 否 排除特定文档

📝 组合查询范例

这是一个典型的组合使用场景，展示了如何在一个查询中同时运用这四种方式：

 1GET /products/_search
 2{
 3  "query": {
 4    "bool": {
 5      "must": [
 6        { "match": { "title": "智能手机" } } 
 7      ],
 8      "filter": [
 9        { "term": { "brand.keyword": "华为" } },
10        { "range": { "price": { "gte": 3000, "lte": 5000 } } }
11      ],
12      "should": [
13        { "match": { "tags": "5G" } },
14        { "match": { "description": "高性能" } }
15      ],
16      "must_not": [
17        { "term": { "status": "offline" } }
18      ],
19      "minimum_should_match": 1
20    }
21  }
22}

查询解读：

  1. must: 搜索标题中包含“智能手机”的商品。
  2. filter: 并且品牌是“华为”，价格在 3000 到 5000 之间。
  3. should: 如果商品的标签包含“5G”或者描述中包含“高性能”，则会获得更高的分数。
  4. must_not: 但绝不能包含状态为“offline”的商品。
  5. minimum_should_match: 强制要求 should 中的条件至少要满足一个。

geo_point 是一种用于存储地理坐标点（经纬度）的专用数据类型，常见于 Elasticsearch 等搜索引擎和数据库中。它不仅仅是存储经纬度数值，更重要的是通过内置的空间索引（如 GeoHash 和 BKD 树）来高效地支持各种地理位置相关的操作。

🎯 核心功能与优势

使用 geo_point 类型而非普通字符串或数字来存储坐标，可以解锁以下强大功能：

  • 高效的地理查询：快速查找特定范围内的点，例如“查找我周围 5 公里内的所有餐厅”。
  • 地理聚合：将地图划分为网格进行统计分析，例如生成“热力图”或统计不同区域的数据密度。
  • 距离排序：根据与某个中心点的距离对结果进行排序，例如“按距离从近到远显示”。
  • 相关性打分：将距离作为因素之一，影响搜索结果的相关性得分。

📝 数据录入格式范例

geo_point 类型非常灵活，支持多种格式录入数据。以下示例均以 Elasticsearch 为例，假设字段名为 location。

格式类型 范例 关键说明
对象 (Object) {"lat": 39.9, "lon": 116.4} 最清晰、最不易出错的格式，推荐优先使用。
字符串 (String) "39.9,116.4" 顺序为“纬度,经度”，中间用逗号分隔。
数组 (Array) [116.4, 39.9] 顺序为“经度,纬度”，与 GeoJSON 标准一致，需特别注意。
Geohash "wx4g0ec1" 一种将经纬度编码成字符串的算法，精度越高字符串越长。
WKT (Well-Known Text) "POINT(116.4 39.9)" 一种标准的文本表示法，注意空格分隔，顺序为“经度 纬度”。

💡 重要注意事项与常见陷阱

在使用 geo_point 时，有几个关键点需要特别注意，以避免出现难以排查的错误：

  1. 经纬度顺序陷阱 这是最常见的错误来源。不同的格式和系统对经纬度的顺序要求不同。

    • 数组格式 [lon, lat]：经度在前。
    • 字符串格式 "lat,lon"：纬度在前。
    • 许多前端地图库（如 Leaflet）使用 [lat, lon] 顺序，而 GeoJSON 和 Elasticsearch 的数组格式使用 [lon, lat]。在数据流转过程中务必进行校验和转换。

  2. 数值范围校验 必须确保经纬度值在有效范围内，否则可能导致数据错误或静默失败。

    • 纬度 (lat)：范围是 [-90, 90]。
    • 经度 (lon)：范围是 [-180, 180]。 建议在数据写入前增加校验逻辑，或在索引映射中设置 ignore_malformed: false 来主动拒绝非法数据。

  3. 性能与功能优势 切勿将坐标存储为普通字符串（如 "39.9,116.4"）或两个独立的数字字段。这样做会失去所有地理空间索引的优势，导致查询、聚合和排序操作性能极差，甚至无法实现。
