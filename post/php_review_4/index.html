2025-06-04 · 6 分钟阅读 · PHP  ·
分享到:

在 PHP 中，配置项（INI 配置项）具有不同的作用域（INI Scope），这决定了该配置项可以在哪些地方被修改。PHP 定义了以下几种作用域常量（定义在 php.ini 的上下文中）：

1. PHP_INI_USER

  • 含义：可以在用户脚本中通过 ini_set() 动态修改。
  • 可修改位置：
    • .htaccess（Apache）
    • httpd.conf 中的 <Directory>、<Location> 等（如果允许）
    • PHP 脚本中使用 ini_set()
  • 示例配置项：display_errors、error_reporting、memory_limit（部分版本）

2. PHP_INI_PERDIR

  • 含义：可以在目录层级的配置文件中设置，但不能在运行时通过 ini_set() 修改。
  • 可修改位置：
    • php.ini
    • .htaccess（Apache）
    • httpd.conf 中的 <Directory>、<VirtualHost> 等
  • 不可在 PHP 脚本中用 ini_set() 修改
  • 示例配置项：upload_max_filesize、post_max_size

3. PHP_INI_SYSTEM

  • 含义：只能在系统级配置文件中设置，对整个 PHP 环境生效。
  • 可修改位置：
    • php.ini
    • Web 服务器主配置（如 Apache 的 httpd.conf 中的 PHPIniDir 或 php_admin_value）
  • 不可在 .htaccess 或 PHP 脚本中修改
  • 示例配置项：extension_dir、safe_mode（已废弃）、enable_dl

4. PHP_INI_ALL

  • 含义：等价于 PHP_INI_USER | PHP_INI_PERDIR | PHP_INI_SYSTEM，即所有地方都可以设置。
  • 实际上，这是某些配置项的“宽松”作用域。
  • 可在 PHP 脚本中用 ini_set() 修改
  • 示例配置项：date.timezone（在较新版本中）

⚠️ 注意：不是所有文档都明确列出每个配置项的作用域，但可以通过 phpinfo() 或 ini_get_all() 查看。

如何查看某个配置项的作用域？

使用 ini_get_all() 函数：

1$config = ini_get_all();
2var_dump($config['display_errors']);

输出可能类似：

1array(3) {
2  ["global_value"] => string(1) "1"
3  ["local_value"]  => string(1) "1"
4  ["access"]       => int(7)  // 7 = PHP_INI_ALL (1+2+4)
5}

其中 access 字段的值对应：

  • 1 = PHP_INI_USER
  • 2 = PHP_INI_PERDIR
  • 4 = PHP_INI_SYSTEM
  • 组合值如 7 表示三者皆可（即 PHP_INI_ALL）

总结：是否可以在 PHP 脚本中修改？

作用域 是否可用 ini_set() 修改？
PHP_INI_USER ✅ 是
PHP_INI_PERDIR ❌ 否
PHP_INI_SYSTEM ❌ 否
PHP_INI_ALL ✅ 是（因为包含 USER）

💡 提示：即使某个配置项理论上允许运行时修改（如 memory_limit），也可能受 SAPI（如 CLI vs FPM）或服务器安全策略（如 disable_functions、open_basedir）限制。

如需进一步确认某个配置项能否动态修改，建议：

  1. 查阅 PHP 官方文档
  2. 使用 ini_get_all() 检查其 access 值
  3. 在实际环境中测试 ini_set() 是否生效

在 PHP-FPM（FastCGI Process Manager）中，进程数量的控制是通过其配置文件（通常是 www.conf 或 pool.d/*.conf）中的几个关键参数来实现的。这些参数决定了 FPM 如何管理子进程（worker processes），从而影响性能、内存使用和并发处理能力。

📌 核心配置项（位于 php-fpm.conf 或 pool.d/www.conf）

1. pm（Process Manager）

决定进程管理器的模式，有三种：

模式 说明
static 固定数量的子进程，始终运行。
dynamic 动态调整子进程数量（最常用）。
ondemand 按需启动子进程（空闲时可完全关闭）。

2. 各模式下的关键参数

✅ pm = static

  • pm.max_children：必须设置，固定子进程数量。 
    1pm = static
2pm.max_children = 50

    所有 50 个进程常驻内存，适合高负载、稳定流量场景。

✅ pm = dynamic（推荐用于大多数 Web 应用）

  • pm.max_children：最大子进程数（硬上限）。
  • pm.start_servers：启动时创建的进程数。
  • pm.min_spare_servers：空闲进程的最小数量（低于此值则创建新进程）。
  • pm.max_spare_servers：空闲进程的最大数量（高于此值则杀掉多余进程）。

示例：

1pm = dynamic
2pm.max_children = 100
3pm.start_servers = 20
4pm.min_spare_servers = 10
5pm.max_spare_servers = 30

💡 建议：

  • pm.min_spare_servers ≤ pm.start_servers ≤ pm.max_spare_servers
  • pm.max_spare_servers < pm.max_children

✅ pm = ondemand

  • pm.max_children：最大子进程数。
  • pm.process_idle_timeout：空闲进程在被杀死前等待的时间（如 10s）。
  • 启动时不创建任何子进程，有请求才创建。

示例：

1pm = ondemand
2pm.max_children = 50
3pm.process_idle_timeout = 10s

适合低流量或开发环境，节省内存，但首次请求有延迟。

🔍 如何合理设置 pm.max_children？

这是最关键的参数，设置不当会导致：

  • 太小 → 请求排队、502/504 错误
  • 太大 → 内存耗尽、系统 OOM

计算公式（经验法则）：

1max_children ≈ 总可用内存 / 每个 PHP-FPM 进程平均内存

步骤：

  1. 查看单个 FPM 进程内存占用（单位 KB）：

    1ps --no-headers -o "rss,cmd" -C php-fpm | awk '{ sum+=$1 } END { printf("%d\n", sum/NR/1024) }'

    或更精确地：

    1ps -ylC php-fpm --sort:rss

  2. 假设：

    • 服务器分配给 PHP-FPM 的内存：2GB = 2048 MB
    • 每个进程平均占用：80 MB

  3. 则：

    1max_children = 2048 / 80 ≈ 25

⚠️ 注意：需为系统和其他服务（如 MySQL、Nginx）预留内存！

🛠 其他相关优化参数

参数 说明
pm.max_requests 每个子进程处理多少请求后重启（防止内存泄漏），建议 500~1000
request_terminate_timeout 单个请求最大执行时间（避免卡死进程）
rlimit_files 文件描述符限制（高并发时需调大）

示例：

1pm.max_requests = 500
2request_terminate_timeout = 30s

🔄 重载配置

修改后需重载 PHP-FPM（不要 restart，避免中断服务）：

1# systemd 系统
2sudo systemctl reload php-fpm
3
4# 或发送信号
5sudo kill -USR2 $(cat /run/php-fpm.pid)

📊 监控建议

  • 使用 pm.status_path 启用状态页（配合 Nginx）： 
    1pm.status_path = /status
    访问后可看到 active processes, idle processes, max children reached 等关键指标。
  • 若 max children reached = yes 频繁出现，说明 max_children 太小。

✅ 总结

场景 推荐模式 关键参数
高并发、稳定流量 static pm.max_children
通用 Web 应用 dynamic max_children, min/max_spare_servers
低流量/开发环境 ondemand max_children, process_idle_timeout

合理配置 PHP-FPM 进程数，能显著提升系统稳定性与资源利用率。务必结合实际内存和业务负载进行调优。

下面是一个完整的配置过程，用于在 PHP-FPM + Nginx 环境中启用 pm.status_path 状态页，并通过 Nginx 安全地暴露访问路径（例如 /fpm-status），同时限制仅允许本地或特定 IP 访问。

✅ 一、前提条件

  • 已安装并运行 PHP-FPM（如 php8.1-fpm、php8.2-fpm 等）
  • 使用 Nginx 作为 Web 服务器
  • 配置文件路径示例（根据系统可能略有不同）：
    • PHP-FPM 池配置：/etc/php/{version}/fpm/pool.d/www.conf
    • Nginx 站点配置：/etc/nginx/sites-available/default 或自定义站点

✅ 二、步骤 1：启用 PHP-FPM 状态页

编辑你的 FPM 池配置文件（通常是 www.conf）：

1sudo nano /etc/php/8.2/fpm/pool.d/www.conf

⚠️ 将 8.2 替换为你实际的 PHP 版本。

找到以下行（通常被注释）并取消注释并修改：

1; 启用状态页路径
2pm.status_path = /fpm-status
3
4; （可选）启用 ping 路径，用于健康检查
5ping.path = /fpm-ping

💡 路径名可以自定义，但建议不要使用常见路径（如 /status）以防被扫描。

保存并退出。

✅ 三、步骤 2：重载 PHP-FPM

让配置生效（不要 restart，避免中断服务）：

1sudo systemctl reload php8.2-fpm

如果你不确定服务名，可用 systemctl list-units | grep fpm 查看。

✅ 四、步骤 3：配置 Nginx 转发状态请求到 PHP-FPM

编辑你的 Nginx 站点配置文件，例如：

1sudo nano /etc/nginx/sites-available/default

在 server { ... } 块中添加以下 location 块：

 1# PHP-FPM 状态页（仅限本地访问）
 2location ~ ^/(fpm-status|fpm-ping)$ {
 3    access_log off;
 4    allow 127.0.0.1;
 5    allow ::1;               # IPv6 localhost
 6    # allow 192.168.1.0/24;  # 如需允许内网，取消注释并修改
 7    deny all;
 8
 9    include fastcgi_params;
10    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
11    fastcgi_pass unix:/run/php/php8.2-fpm.sock;  # 根据你的 sock 路径调整
12}

🔍 关键点说明：

  • fastcgi_pass 必须与 PHP-FPM 监听的 socket 或 TCP 地址一致。 > - 查看 www.conf 中的 listen = /run/php/php8.2-fpm.sock 或 127.0.0.1:9000
  • 默认只允许 127.0.0.1（本机）访问，防止信息泄露。

✅ 五、步骤 4：测试配置 & 重载 Nginx

  1. 检查 Nginx 配置语法：

    1sudo nginx -t

    确保输出为 syntax is ok 和 test is successful。

  2. 重载 Nginx：

    1sudo systemctl reload nginx

✅ 六、步骤 5：验证状态页是否生效

从服务器本机访问：

1curl http://127.0.0.1/fpm-status

你应该看到类似输出：

 1pool:                 www
 2process manager:      dynamic
 3start time:           16/Jan/2026:14:20:00 +0800
 4start since:          120
 5accepted conn:        45
 6listen queue:         0
 7max listen queue:     0
 8listen queue len:     0
 9idle processes:       2
10active processes:     1
11total processes:      3
12max active processes: 3
13max children reached: 0
14slow requests:        0

也可以访问 /fpm-ping，应返回 pong。

❌ 如果返回 403，说明 IP 被拒绝 —— 检查 allow 规则。
❌ 如果返回 404 或 502，检查 fastcgi_pass 是否匹配 PHP-FPM 的 listen 地址。

✅ 七、（可选）格式化输出（JSON）

PHP-FPM 状态页支持 JSON 格式，便于监控脚本解析：

1curl http://127.0.0.1/fpm-status?json

返回：

1{
2  "pool": "www",
3  "process manager": "dynamic",
4  "start time": 1771234800,
5  "start since": 120,
6  "accepted conn": 45,
7  ...
8}

其他参数：

  • ?full：显示每个子进程的详细信息
  • ?json&full：JSON + 详细信息

✅ 八、安全提醒

  • 切勿对公网开放 /fpm-status！它会暴露进程数、内存、请求统计等敏感信息。
  • 如需远程监控，建议：
    • 通过 SSH 隧道访问
    • 或配合 Prometheus + Exporter（如 phpfpm_exporter）
    • 或使用防火墙限制 IP

🎯 总结

步骤 操作
1 在 www.conf 中设置 pm.status_path = /fpm-status
2 重载 PHP-FPM
3 在 Nginx 中添加 location 转发规则，限制 IP
4 重载 Nginx
5 用 curl http://127.0.0.1/fpm-status 验证

完成以上步骤后，你就可以安全地监控 PHP-FPM 的运行状态了！

使用 PHP OpenSSL 扩展生成 RSA 2048 位密钥对，并实现短字符串加密/解密的完整可运行示例，包含关键说明、错误处理、长度限制提示及长文本解决方案建议。

🔒 完整可运行示例（短字符串）

 1<?php
 2/**
 3 * RSA 2048 位密钥生成 + 公钥加密 / 私钥解密示例
 4 * 注意：仅适用于短字符串（≤245字节），长文本需用混合加密（见文末说明）
 5 */
 6
 7// ============= 1. 生成 RSA 密钥对 =============
 8$config = [
 9    'digest_alg'       => 'sha512',
10    'private_key_bits' => 2048,
11    'private_key_type' => OPENSSL_KEYTYPE_RSA,
12];
13
14// 生成私钥资源
15$privateKeyRes = openssl_pkey_new($config);
16if (!$privateKeyRes) {
17    die("❌ 密钥生成失败: " . openssl_error_string() . "\n");
18}
19
20// 导出 PEM 格式私钥字符串
21openssl_pkey_export($privateKeyRes, $privateKeyPem);
22
23// 从私钥资源提取公钥 PEM 字符串
24$pubKeyDetails = openssl_pkey_get_details($privateKeyRes);
25$publicKeyPem = $pubKeyDetails['key'];
26
27// 释放资源（可选）
28openssl_pkey_free($privateKeyRes);
29
30echo "✅ 密钥对生成成功！\n";
31echo "🔐 私钥 (PEM):\n" . wordwrap($privateKeyPem, 64, "\n", true) . "\n";
32echo "🔓 公钥 (PEM):\n" . wordwrap($publicKeyPem, 64, "\n", true) . "\n\n";
33
34// ============= 2. 加密（公钥） =============
35$original = "Hello RSA! 这是测试消息。"; // 中文需注意编码，UTF-8 安全
36echo "📝 原始消息: $original (长度: " . strlen($original) . " 字节)\n";
37
38// 检查长度（2048位RSA + PKCS1 padding 最大245字节）
39if (strlen($original) > 245) {
40    die("❌ 消息过长！RSA 2048 仅支持 ≤245 字节（使用 PKCS1 padding）。\n" .
41        "💡 建议：使用混合加密（RSA+AES）处理长文本。\n");
42}
43
44$encryptedRaw = '';
45if (!openssl_public_encrypt($original, $encryptedRaw, $publicKeyPem, OPENSSL_PKCS1_PADDING)) {
46    die("❌ 加密失败: " . openssl_error_string() . "\n");
47}
48
49$encryptedBase64 = base64_encode($encryptedRaw);
50echo "🔒 加密后 (Base64): " . $encryptedBase64 . "\n\n";
51
52// ============= 3. 解密（私钥） =============
53$decrypted = '';
54$encryptedBinary = base64_decode($encryptedBase64);
55
56if (!openssl_private_decrypt($encryptedBinary, $decrypted, $privateKeyPem, OPENSSL_PKCS1_PADDING)) {
57    die("❌ 解密失败: " . openssl_error_string() . "\n");
58}
59
60echo "✅ 解密结果: $decrypted\n";
61echo "matchCondition: " . ($original === $decrypted ? "✅ 一致" : "❌ 不一致") . "\n";
62
63// ============= 4. （可选）保存密钥到文件 =============
64// file_put_contents('private.key', $privateKeyPem);
65// file_put_contents('public.key', $publicKeyPem);
66// echo "\n🔑 密钥已保存至 private.key / public.key\n";
67?>

⚠️ 重要说明与最佳实践

🔑 密钥安全

  • 私钥必须严格保密，切勿提交到 Git 或暴露在前端。
  • 生产环境建议使用 chmod 600 private.key 限制文件权限。
  • 密钥应存储在安全位置（如环境变量、密钥管理服务）。

📏 长度限制（关键！）

密钥长度 填充方式 最大加密数据长度
2048 位 PKCS1 (默认) 245 字节
2048 位 OAEP (更安全) 214 字节
  • 示例中使用 OPENSSL_PKCS1_PADDING（兼容性好），如需更高安全性可改用 OPENSSL_PKCS1_OAEP_PADDING（需加解密端一致）。
  • 中文/UTF-8 字符：strlen() 按字节计算，中文通常占 3 字节，需特别注意长度。

🌐 长文本解决方案（强烈推荐）

RSA 不适合直接加密长文本。工业级做法是 混合加密：

  1. 生成随机 AES 密钥（如 256 位）
  2. 用 RSA 公钥加密 AES 密钥
  3. 用 AES 加密实际数据
  4. 传输：[加密的AES密钥] + [AES加密的数据]
  5. 接收方用 RSA 私钥解密 AES 密钥，再用 AES 解密数据

✅ 优势：安全、高效、无长度限制
📚 参考：PHP openssl_encrypt() / openssl_decrypt() + random_bytes()

🛠️ 常见问题排查

问题 解决方案
openssl_pkey_new 失败 检查 php.ini 中 openssl.cnf 路径是否正确（Windows 常见）
中文乱码 确保脚本、数据、终端均为 UTF-8 编码
解密后为空 检查 Base64 编解码、填充方式是否一致

💡 总结

  • ✅ 本示例适用于短消息（如 token、密钥片段）的加解密验证。
  • 🔒 切勿用 RSA 直接加密用户密码、长文本——应使用哈希（密码）或混合加密（长数据）。
  • 🌱 新项目建议结合 libsodium（PHP 7.2+ 内置）使用更现代的加密方案（如 sodium_crypto_box）。
