Pod资源对象

Pod是Kubernetes中最小的调度单元，一个Pod可以包含一个或者多个容器，这些容器运行在同一个节点上，从而可以共享网络、存储以及其他资源

Kubernetes Pause 容器作用总结

一、什么是 Pause 容器

Pause 容器（Infrastructure Container，基础设施容器）是 Kubernetes 在创建 Pod 时自动启动的一个特殊容器。

它本身不承载业务，只运行一个极其简单的程序：

1for (;;) {
2    pause();
3}

镜像通常为：

1registry.k8s.io/pause:3.10

每个 Pod 中都会存在一个 Pause 容器，其主要职责是：

为整个 Pod 提供并维护共享的 Namespace 环境，使 Pod 内多个容器能够像运行在同一台主机上一样进行协作。

二、Pause 容器在 Pod 中的位置

例如：

1Pod
2├── pause
3├── nginx
4├── redis
5└── sidecar

其中：

  • Pause 容器是 Pod 的基础容器；
  • nginx、redis、sidecar 等业务容器共享 Pause 容器创建的 Namespace。

结构如下：

 1                Pod
 2                 │
 3         ┌───────┴───────┐
 4         │               │
 5      Pause容器      Namespace
 6         │
 7 ┌───────┼──────────────────┐
 8 │       │         │        │
 9Network  IPC      UTS      PID
10 │       │         │        │
11 ├────────────┬─────────────┤
12 │            │             │
13nginx       redis        sidecar

三、Pause 容器的主要作用

1、维护 Pod 的网络命名空间（最核心）

功能

Pause 容器创建并保存：

  • Network Namespace
  • Pod IP
  • 路由表
  • 网卡（eth0）
  • iptables 规则

其它业务容器都加入该网络空间。

示意图：

1Pod
2│
3├── pause
4│      ↑
5│      Network Namespace
6│
7├── nginx
8├── redis
9└── sidecar

实现效果

Pod 内所有容器共享：

  • IP 地址
  • MAC 地址
  • Port
  • Route
  • localhost

例如：

nginx 容器可以直接访问：

1curl localhost:6379

连接 redis。

2、保证 Pod IP 不变

假设：

1Pod
2├── pause
3├── nginx
4└── sidecar

如果 nginx 容器异常退出：

1pause（仍然存在）
2sidecar（继续运行）
3nginx（重新创建）

由于 Pause 容器未退出：

  • Pod IP 不变；
  • 网络 Namespace 不会销毁；
  • Service Endpoint 不需要更新；
  • Pod 内容器间通信不会中断。

实现了：

容器生命周期与 Pod 生命周期解耦。

3、提供共享 IPC Namespace

Pause 容器维护：

  • Shared Memory
  • Message Queue
  • Semaphore

因此 Pod 内多个容器可以进行进程间通信。

共享内容包括：

1/dev/shm
2消息队列
3信号量

适用于：

  • sidecar 模式；
  • 数据共享；
  • 高性能进程通信。

4、提供统一的 Hostname（UTS Namespace）

Pod 内所有容器共享：

1hostname

得到的结果完全一致：

1mypod

这是因为：

1UTS Namespace

由 Pause 容器负责创建和维护。

5、提供 PID Namespace（可选）

当开启：

1spec:
2  shareProcessNamespace: true

时，Pod 内所有容器共享进程空间。

例如：

1Pod
2├── pause
3├── nginx
4└── fluentd

执行：

1ps -ef

结果：

1PID COMMAND
2
31 pause
48 nginx
535 fluentd

此时：

Pause 容器成为整个 Pod 的：

1PID 1

能够看到其它容器中的进程。

6、回收僵尸进程

Linux 中 PID 1 有特殊职责：

  • 收养孤儿进程；
  • 回收 Zombie Process（僵尸进程）。

Pause 容器作为：

1PID 1

能够：

  • wait()
  • waitpid()

处理 SIGCHLD 信号。

避免：

1Zombie Process

长期占用系统资源。

7、作为 Pod 生命周期的锚点

整个 Pod 是否存在，本质上取决于：

1Pause 容器是否存在

删除 Pod 时：

11. 停止业务容器
22. 停止 sidecar
33. 最后停止 Pause 容器

当 Pause 容器退出：

1Network Namespace 删除
2IPC Namespace 删除
3UTS Namespace 删除
4PID Namespace 删除

Pod 生命周期结束。

8、实现 localhost 通信

例如：

1Pod
2├── nginx:80
3└── sidecar:8080

sidecar 中：

1curl localhost:80

即可访问 nginx。

原因：

两个容器共享：

1127.0.0.1

而共享 localhost 的基础，就是 Pause 容器维护的 Network Namespace。

四、为什么需要 Pause 容器

如果没有 Pause 容器

假设 nginx 是第一个容器：

1nginx
2redis
3sidecar

若 nginx 崩溃：

1nginx退出
2↓
3Namespace销毁
4↓
5Pod网络消失
6↓
7其它容器全部异常

整个 Pod 会受到影响。

引入 Pause 容器后

1pause
2nginx
3redis
4sidecar

即使：

1nginx退出

也会：

1pause仍存在
2↓
3Namespace继续存在
4↓
5Pod IP不变
6↓
7重新拉起nginx

因此：

Pod 生命周期独立于业务容器生命周期。

五、Pause 容器工作原理

在底层实际上类似：

1docker run pause
2
3docker run --net=container:pause nginx
4
5docker run --net=container:pause redis
6
7docker run --net=container:pause sidecar

共享：

1--net=container:pause
2--ipc=container:pause
3--uts=container:pause

因此：

所有业务容器都是加入 Pause 容器已经创建好的 Namespace。

六、查看 Pause 容器

containerd

查看：

1crictl ps -a

示例：

七、Pause 容器作用汇总表

八、总结

Pause 容器虽然几乎不执行任何业务逻辑，但它是 Kubernetes Pod 的基础设施容器，负责维护整个 Pod 的共享 Namespace 环境。

其核心价值可以概括为：

Pause 容器通过持有 Network、IPC、UTS、PID 等 Namespace，使多个业务容器能够共享同一个运行环境，并实现 Pod 生命周期与容器生命周期的解耦，从而保证 Pod 网络稳定和多容器协同工作。

一句话总结

Pause 容器是 Kubernetes Pod 的“根容器”，负责维护 Pod 的共享运行环境，是整个 Pod 生命周期和网络空间的载体。

Kubernetes Pod 中 command 和 args 字段详解

一、概述

在 Kubernetes 中，Pod 内每个容器都可以通过：

1command:
2args:

两个字段来定义容器启动时执行的命令及其参数。

它们的作用相当于 Docker 中的：

1command → ENTRYPOINT
2args    → CMD

用于：

  • 指定容器启动程序；
  • 覆盖镜像中的默认启动命令；
  • 传递启动参数；
  • 执行初始化脚本；
  • 实现定制化运行。

二、基本语法

1spec:
2  containers:
3  - name: nginx
4    image: nginx
5    command:
6    - ...
7    args:
8    - ...

其中：

  • command

    • 指定容器启动命令；
    • 对应 Docker 的 ENTRYPOINT。

  • args

    • 指定启动参数；
    • 对应 Docker 的 CMD。

三、command 与 args 的对应关系

Dockerfile：

1ENTRYPOINT ["nginx"]
2CMD ["-g","daemon off;"]

对应 Kubernetes：

1command:
2- nginx
3
4args:
5- "-g"
6- "daemon off;"

启动后实际执行：

1nginx -g "daemon off;"

四、command 和 args 与 Docker 的关系

Kubernetes Docker 含义
command ENTRYPOINT 主命令
args CMD 参数
command+args ENTRYPOINT+CMD 最终执行命令

因此：

1command:
2- sleep
3
4args:
5- "3600"

最终执行：

1sleep 3600

五、镜像默认命令

例如：

Dockerfile：

1ENTRYPOINT ["nginx"]
2CMD ["-g","daemon off;"]

当 Pod 中没有指定：

1containers:
2- image: nginx

实际执行：

1nginx -g "daemon off;"

使用：

1kubectl describe pod

可以看到：

1Command:
2Args:

为空。

说明：

使用的是镜像自身定义的 ENTRYPOINT 和 CMD。

六、只指定 args

镜像

Dockerfile：

1ENTRYPOINT ["nginx"]
2CMD ["-g","daemon off;"]

Pod：

1containers:
2- image: nginx
3
4  args:
5  - "-g"
6  - "daemon off;"

实际执行：

1nginx -g "daemon off;"

特点

  • ENTRYPOINT 不变；
  • 只覆盖 CMD。

再例如

镜像：

1ENTRYPOINT ["sleep"]
2CMD ["10"]

Pod：

1args:
2- "3600"

最终：

1sleep 3600

七、只指定 command

Pod：

1containers:
2- image: nginx
3
4  command:
5  - sleep

实际执行：

1sleep

此时：

command 会覆盖：

1ENTRYPOINT

args 会被清空

即：

镜像中的：

1CMD ["-g","daemon off;"]

失效。

八、同时指定 command 和 args

1containers:
2- image: nginx
3
4  command:
5  - sleep
6
7  args:
8  - "3600"

实际执行：

1sleep 3600

此时：

镜像中的：

1ENTRYPOINT
2CMD

全部失效。

完全以 Pod 定义为准。

九、四种情况总结

假设镜像：

1ENTRYPOINT ["nginx"]
2CMD ["-g","daemon off;"]
command args 最终执行
不指定 不指定 nginx -g daemon off;
不指定 指定 nginx + args
指定 不指定 command
指定 指定 command + args

总结：

1command → 覆盖 ENTRYPOINT
2args → 覆盖 CMD

十、数组形式

推荐：

1command:
2- sleep
3
4args:
5- "3600"

等价于：

1sleep 3600

因为：

1command:
2- sleep
3- 3600

实际执行：

1sleep 3600

但推荐：

1command:
2- sleep
3
4args:
5- "3600"

职责更清晰。

十一、执行 Shell 命令

错误写法：

1command:
2- "echo hello && sleep 3600"

因为：

1command:
2- /bin/sh

4
5args:

2echo hello
3sleep 3600

Kubernetes 容器探针（Probe）详解

一、什么是 Probe（探针）

Probe（探针）是 Kubernetes 用来检测容器运行状态的一种机制。

Kubelet 通过定期执行探测操作，判断：

  • 容器是否已经启动完成；
  • 容器是否正常运行；
  • 容器是否能够对外提供服务。

根据探测结果，Kubernetes 可以：

  • 重启异常容器；
  • 控制 Pod 是否加入 Service；
  • 延迟业务流量进入；
  • 提高应用的高可用性。

二、Probe 类型

Kubernetes 共支持三种探针：

探针类型 字段 作用
Startup Probe startupProbe 判断容器是否启动完成
Liveness Probe livenessProbe 判断容器是否存活
Readiness Probe readinessProbe 判断容器是否可以提供服务

整体关系：

1容器启动
2     ↓
3startupProbe
4     ↓
5启动成功
6     ↓
7livenessProbe + readinessProbe
8     ↓
9正常提供服务

三、Startup Probe（启动探针）

作用

用于判断：

容器是否已经完成启动。

适用于：

  • Java 应用
  • Spring Boot
  • Elasticsearch
  • Kafka
  • 数据库

等启动时间较长的应用。

工作流程

1容器启动
2     ↓
3startupProbe 持续检查
4     ↓
5成功
6     ↓
7开始执行：
8    livenessProbe
9    readinessProbe

在 Startup Probe 成功之前：

1Liveness Probe
2Readiness Probe

不会执行。

探测失败

若超过允许次数：

1startupProbe failure
2↓
3容器重启

示例

1startupProbe:
2  httpGet:
3    path: /health
4    port: 8080
5
6  initialDelaySeconds: 10
7  periodSeconds: 5
8  failureThreshold: 30

最大等待时间：

130 × 5 = 150 秒

如果 150 秒内应用未启动成功：

1Container Restart

适用场景

Spring Boot

启动需要：

11~2分钟

若直接配置：

1livenessProbe

可能：

1应用还未启动
2↓
3探测失败
4↓
5容器重启
6↓
7无限循环

因此：

推荐：

1startupProbe

先等待应用启动完成。

四、Liveness Probe（存活探针）

作用

判断：

容器是否仍然活着。

如果探测失败：

1Restart Container

由 Kubelet 自动重启容器。

工作流程

 1运行中
 2    ↓
 3livenessProbe
 4    ↓
 5失败
 6    ↓
 7达到 failureThreshold
 8    ↓
 9Kill Container
10    ↓
11重新创建容器

典型故障

死锁

程序：

1线程死锁

虽然：

1进程存在

但：

1无法响应请求

Liveness Probe：

1检测失败
2↓
3自动重启

恢复服务。

示例

1livenessProbe:
2  httpGet:
3    path: /health
4    port: 8080
5
6  initialDelaySeconds: 30
7  periodSeconds: 10

每：

110 秒

检测一次。

探测失败

达到：

1failureThreshold: 3

后：

1连续失败 3 次
2↓
3重启容器

适用场景

适合：

  • Web 服务
  • API 服务
  • Redis
  • Nginx
  • MySQL

用于发现：

  • 死锁
  • 卡死
  • 无响应

五、Readiness Probe（就绪探针）

作用

判断：

当前容器是否能够对外提供服务。

Readiness Probe 失败：

不会重启容器

只会：

1Pod 从 Endpoint 中移除

停止接收流量。

工作流程

1Readiness Probe
2        ↓
3成功
4        ↓
5加入 Service
6        ↓
7接收流量

失败：

1Readiness Probe
2       ↓
3失败
4       ↓
5Endpoint 删除
6       ↓
7不再接收流量

容器仍然运行。

示例

1readinessProbe:
2  httpGet:
3    path: /ready
4    port: 8080
5
6  initialDelaySeconds: 20
7  periodSeconds: 5

适用场景

数据库连接未建立

应用虽然启动：

1Java Process

但是：

1数据库连接池初始化中

此时：

1Readiness Probe 返回失败

Pod 不会加入：

1Service Endpoints

等数据库连接完成：

1Probe Success

自动接收流量。

六、三种 Probe 的区别

探针 检测内容 失败后行为
Startup Probe 是否启动完成 重启容器
Liveness Probe 是否存活 重启容器
Readiness Probe 是否可提供服务 从 Endpoint 中移除

七、执行顺序

没有 Startup Probe

1Container Start
2      ↓
3Liveness Probe
4      ↓
5Readiness Probe

有 Startup Probe

1Container Start
2       ↓
3Startup Probe
4       ↓
5成功
6       ↓
7Liveness Probe
8       ↓
9Readiness Probe

即：

1startupProbe
2优先级最高

成功之后：

1livenessProbe
2readinessProbe

才开始工作。

八、Probe 的探测方式

Kubernetes 支持四种探测方式。

类型 字段 说明
Exec exec 执行命令
HTTP httpGet HTTP 请求
TCP tcpSocket TCP 端口检查
gRPC grpc gRPC 服务检查

九、Exec 探针

通过执行命令判断。

例如：

1livenessProbe:
2  exec:
3    command:
4    - cat
5    - /tmp/healthy

若：

1exit code = 0

成功。

否则：

1失败

适用场景

数据库：

1mysqladmin ping

Redis：

1redis-cli ping

十、HTTP 探针

最常用。

例如：

1readinessProbe:
2  httpGet:
3    path: /health
4    port: 8080

返回：

1200~399

视为成功。

返回：

14xx
25xx

视为失败。

Spring Boot

1management:

十一、TCP 探针

检查端口是否可连接。

例如：

1livenessProbe:
2  tcpSocket:
3    port: 3306

如果：

1connect()
2成功

则探测成功。

适用

  • MySQL
  • Redis
  • Kafka

十二、gRPC 探针

K8s 1.24 后支持。

例如：

1livenessProbe:
2  grpc:
3    port: 50051

用于：

  • gRPC 微服务

十三、公共参数

所有 Probe 都支持：

参数 默认值 含义
initialDelaySeconds 0 延迟多久开始探测
periodSeconds 10 多久检测一次
timeoutSeconds 1 超时时间
successThreshold 1 连续成功次数
failureThreshold 3 连续失败次数

最大失败时间

例如：

1periodSeconds: 10
2failureThreshold: 3

则：

1连续失败 30 秒

后：

  • Liveness → 重启
  • Readiness → 下线

一句话总结

Startup Probe 用于判断“是否启动完成”，Liveness Probe 用于判断“是否还活着”，Readiness Probe 用于判断“是否可以接收流量”；而 Exec、HTTP、TCP、gRPC 则是实现这些探针的具体检测方式。 🚀
