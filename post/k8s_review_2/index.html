Kubernetes 集群的部署方式很多，但从实际应用和学习角度来看，主要可以归纳为 三种部署方式：

1. 二进制部署（Binary Installation）

部署方式

手动安装 Kubernetes 各组件的二进制文件：

  • kube-apiserver
  • kube-controller-manager
  • kube-scheduler
  • kubelet
  • kube-proxy
  • etcd
  • Container Runtime（Docker、containerd）

特点

优点 缺点
灵活性最高 安装复杂
可深度定制 运维成本高
适合生产环境 升级和维护较麻烦
对组件版本控制精确 学习门槛较高

适用场景

  • 企业生产环境
  • 对集群有特殊定制需求
  • 国产化、离线环境部署
  • 大规模集群

架构示意

 1Master
 2├── etcd
 3├── kube-apiserver
 4├── kube-controller-manager
 5└── kube-scheduler
 6
 7Node
 8├── kubelet
 9├── kube-proxy
10└── containerd(Docker)

2. kubeadm 部署（官方推荐）

部署方式

利用官方工具 kubeadm 自动完成集群初始化。

常用命令：

1# 初始化 Master
2kubeadm init
3
4# Node 加入集群
5kubeadm join ...

特点

优点 缺点
部署简单 灵活性略低
官方支持 部分组件配置受限制
升级方便 不适合高度定制
文档丰富 -

适用场景

  • 学习 Kubernetes
  • 中小型生产环境
  • 测试环境
  • 企业常规部署

架构

 1Master
 2├── etcd
 3├── kube-apiserver
 4├── kube-controller-manager
 5└── kube-scheduler
 6
 7Node
 8├── kubelet
 9├── kube-proxy
10└── container runtime

目前绝大多数公司（约 80% 以上）都采用 kubeadm 部署或基于 kubeadm 二次封装。

3. 自动化部署工具（第三方发行版）

利用现成的 Kubernetes 安装平台或发行版快速部署。

常见方案：

Rancher + RKE/RKE2

  • 安装简单
  • Web UI 管理
  • 企业级功能丰富

Kubespray

  • 基于 Ansible
  • 支持高可用
  • 大规模集群部署

K3s

  • 轻量级 Kubernetes
  • IoT、边缘计算

Minikube

  • 单节点学习环境

Kind

  • Docker 容器模拟 Kubernetes

特点

优点 缺点
一键部署 对底层了解较少
运维简单 依赖第三方工具
支持 HA 可控性稍差

适用场景

  • 快速搭建
  • 开发测试
  • 企业私有云平台
  • 边缘计算

我的 Kubernetes 版本是 v1.28，准备创建kubernetes-dashboard的登陆账号，推荐采用 TokenRequest API 创建长期 Token。默认生成的 Token 是有过期时间的，可以通过指定较长的有效期来实现。

1. 创建 ServiceAccount

1apiVersion: v1
2kind: ServiceAccount
3metadata:
4  name: admin-user
5  namespace: kubernetes-dashboard

创建：

1kubectl apply -f admin-user.yaml

2. 授予集群管理员权限

创建 ClusterRoleBinding：

 1apiVersion: rbac.authorization.k8s.io/v1
 2kind: ClusterRoleBinding
 3metadata:
 4  name: admin-user
 5roleRef:
 6  apiGroup: rbac.authorization.k8s.io
 7  kind: ClusterRole
 8  name: cluster-admin
 9subjects:
10- kind: ServiceAccount
11  name: admin-user
12  namespace: kubernetes-dashboard

创建：

1kubectl apply -f admin-user-binding.yaml

3. 生成长期 Token

方式一：有效期一年（推荐）

1kubectl create token admin-user \
2  -n kubernetes-dashboard \
3  --duration=8760h

其中：

18760h = 365 天

生成出来的 Token 可以直接用于登录 Dashboard。

方式二：有效期十年

1kubectl create token admin-user \
2  -n kubernetes-dashboard \
3  --duration=87600h

即：

187600h = 3650 天 ≈ 10 年

不过需要注意：

Token 的最大有效期受 apiserver 参数 --service-account-max-token-expiration 限制。

查看：

1ps -ef | grep kube-apiserver

如果没有配置：

1--service-account-max-token-expiration

默认可能只允许 1 小时或 24 小时（不同版本略有差异）。

4. 真正永久不过期（推荐 Dashboard 使用）

创建一个 Secret，让 Kubernetes 自动生成 Legacy Token。

admin-user-secret.yaml

1apiVersion: v1
2kind: Secret
3metadata:
4  name: admin-user-token
5  namespace: kubernetes-dashboard
6  annotations:
7    kubernetes.io/service-account.name: admin-user
8type: kubernetes.io/service-account-token

创建：

1kubectl apply -f admin-user-secret.yaml

查看 Secret：

1kubectl get secret -n kubernetes-dashboard

输出：

1NAME               TYPE                                  DATA
2admin-user-token   kubernetes.io/service-account-token   3

获取 Token：

1kubectl get secret admin-user-token \
2-n kubernetes-dashboard \
3-o jsonpath='{.data.token}' | base64 -d

或者：

1kubectl describe secret admin-user-token -n kubernetes-dashboard

这种方式生成的是：

  • ✅ 长期有效
  • ✅ Pod 重启不受影响
  • ✅ Dashboard 登录方便
  • ✅ 不需要反复复制 Token
  • ✅ 非常适合实验环境和本地集群

推荐最终方案（适用于你的 Kubernetes v1.28 + Dashboard）

把下面三个 YAML 放到一个文件 dashboard-admin-user.yaml 中：

 1apiVersion: v1
 2kind: ServiceAccount
 3metadata:
 4  name: admin-user
 5  namespace: kubernetes-dashboard
 6---
 7apiVersion: rbac.authorization.k8s.io/v1
 8kind: ClusterRoleBinding
 9metadata:
10  name: admin-user
11roleRef:
12  apiGroup: rbac.authorization.k8s.io
13  kind: ClusterRole
14  name: cluster-admin
15subjects:
16- kind: ServiceAccount
17  name: admin-user
18  namespace: kubernetes-dashboard
19---
20apiVersion: v1
21kind: Secret
22metadata:
23  name: admin-user-token
24  namespace: kubernetes-dashboard
25  annotations:
26    kubernetes.io/service-account.name: admin-user
27type: kubernetes.io/service-account-token

执行：

1kubectl apply -f dashboard-admin-user.yaml

然后获取 Token：

1kubectl get secret admin-user-token \
2-n kubernetes-dashboard \
3-o jsonpath='{.data.token}' | base64 -d

这样得到的 Token 基本可以长期使用，非常适合 Kubernetes Dashboard 登录。🚀

kubectl 命令总体结构

kubectl -o 输出格式汇总

在使用 kubectl get、kubectl describe 等命令查看资源时，可以通过 -o（--output）参数指定输出格式。

例如：

1kubectl get pod nginx -o yaml
2kubectl get svc -o json

下面是 kubectl 常见支持的输出格式汇总：

输出格式 描述
wide 在默认表格基础上显示更多信息，例如 Pod IP、节点名称、镜像等
yaml 以 YAML 格式输出资源完整定义
json 以 JSON 格式输出资源完整定义
name 仅输出资源名称，格式为 <资源类型>/<资源名称>
jsonpath 使用 JSONPath 表达式提取指定字段
jsonpath-file 从文件中读取 JSONPath 表达式进行字段提取
go-template 使用 Go Template 模板输出指定内容
go-template-file 从文件中读取 Go Template 模板
custom-columns 自定义列输出
custom-columns-file 从文件读取自定义列定义
template Go Template 的简写方式（较少使用）
templatefile 从模板文件生成输出（较少使用）

常见输出格式示例

1. wide

默认表格扩展输出

1kubectl get pods -o wide

输出：

1NAME     READY STATUS  IP            NODE
2nginx    1/1   Running 10.244.1.2    node1

适用于：

  • 查看 Pod IP
  • 查看节点位置
  • 查看镜像等附加信息

2. yaml ⭐⭐⭐⭐⭐

输出资源完整 YAML

1kubectl get pod nginx -o yaml

适用于：

  • 导出资源
  • 查看全部配置
  • 编写 YAML 文件

3. json ⭐⭐⭐⭐⭐

输出 JSON 格式

1kubectl get pod nginx -o json

适用于：

  • 程序调用
  • jq 工具处理
  • API 调试

4. name

仅输出资源名称

1kubectl get pods -o name

输出：

1pod/nginx
2pod/mysql

常用于：

1kubectl delete $(kubectl get pods -o name)

5. jsonpath ⭐⭐⭐⭐⭐

提取指定字段

获取 Pod IP

1kubectl get pod nginx \
2-o jsonpath='{.status.podIP}'

输出：

110.244.1.2

获取所有 Pod 名称

1kubectl get pods \
2-o jsonpath='{.items[*].metadata.name}'

输出：

1nginx mysql redis

获取 Node 名称

1kubectl get nodes \
2-o jsonpath='{.items[*].metadata.name}'

6. jsonpath-file

从文件读取 JSONPath

1kubectl get pod nginx \
2-o jsonpath-file=path.txt

path.txt：

1{.status.podIP}

适用于：

  • 长 JSONPath 表达式
  • 自动化脚本

7. custom-columns ⭐⭐⭐⭐

自定义列输出

1kubectl get pods \
2-o custom-columns=\
3NAME:.metadata.name,\
4IP:.status.podIP,\
5NODE:.spec.nodeName

输出：

1NAME    IP           NODE
2nginx   10.244.1.2   node1
3mysql   10.244.2.3   node2

适用于：

  • 运维统计
  • 报表输出

8. custom-columns-file

从文件读取列定义

1kubectl get pods \
2-o custom-columns-file=columns.txt

columns.txt：

1NAME:.metadata.name
2NODE:.spec.nodeName
3IP:.status.podIP

9. go-template ⭐⭐⭐

使用 Go 模板

1kubectl get pods \
2-o go-template='{{range .items}}{{.metadata.name}}{{"\n"}}{{end}}'

输出：

1nginx
2mysql
3redis

适用于：

  • 复杂格式化输出
  • 自动化脚本

10. go-template-file

从模板文件读取

1kubectl get pods \
2-o go-template-file=template.tpl

template.tpl：

1{{range .items}}
2{{.metadata.name}}
3{{end}}

日常最常用的几种格式 ⭐⭐⭐⭐⭐

输出格式 典型用途
-o wide 查看更多资源信息
-o yaml 导出资源定义

-o jsonpath 提取单个字段
-o custom-columns 自定义表格输出

其中：

  • 运维人员最常用： wide、yaml
  • 脚本自动化最常用： json、jsonpath
  • CKA/CKAD 考试最常用： yaml、jsonpath、custom-columns
  • 复杂格式化输出： go-template
