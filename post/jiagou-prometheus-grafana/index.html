十一、PromQL（Prometheus查询语言）

这是Prometheus最重要的知识。

查询全部

1up

查询某实例

1up{instance="localhost:9100"}

查询CPU

1node_cpu_seconds_total

查询Memory

1node_memory_MemAvailable_bytes

rate()

统计增长速度。

例如：

1rate(http_requests_total[5m])

最近5分钟：

每秒请求数。

increase()

1increase(http_requests_total[1h])

最近一小时：

请求增加多少。

sum()

1sum(rate(http_requests_total[5m]))

全部机器。

avg()

1avg(node_load1)

平均负载。

max()

1max(node_memory_usage)

topk()

1topk(5,node_load1)

负载最高5台机器。

by()

1sum by(instance)

按机器统计。

without()

1sum without(cpu)

去掉CPU维度。

二十三、企业最佳实践

1. 统一监控架构

1Exporter
2    │
3    ▼
4Prometheus
5    │
6    ├── Alertmanager（告警）
7    ├── Grafana（展示）
8    └── Remote Write（长期存储，如 Thanos、VictoriaMetrics）

2. 标签设计规范

  • 使用有业务意义的标签，如 job、instance、namespace、pod、service、env（dev/test/prod）、cluster。
  • 避免高基数（High Cardinality）标签，例如用户 ID、订单号、随机 UUID，否则会导致 TSDB 内存和磁盘压力急剧增加。

3. 告警设计原则

  • 症状告警优先于原因告警（例如接口不可用比 CPU 高更重要）。
  • 设置合理持续时间（for: 5m），避免瞬时抖动导致误报。
  • 对告警进行分级（P1/P2/P3）并结合告警抑制与静默策略。

4. 数据保留策略

  • Prometheus 本地仅保留近几周数据（如 15～30 天）。
  • 长期存储建议结合 Thanos、VictoriaMetrics 或 Mimir。

📚 结合学习路线，下一步建议

根据我们之前关于 Kubernetes、Helm、Jenkins、Argo CD 的讨论，建议按下面的顺序继续深入，这样知识体系会比较完整：

  1. Prometheus 架构与 TSDB 原理
  2. PromQL 查询语言（重点）
  3. Exporter 体系（Node、MySQL、Redis、Nginx、Blackbox、kube-state-metrics）
  4. Alertmanager（企业微信/钉钉/邮件告警）
  5. Grafana Dashboard 与 Variables
  6. Kubernetes 监控体系（Prometheus Operator、ServiceMonitor、PodMonitor）
  7. 高可用与长期存储（Thanos、VictoriaMetrics、Mimir）
  8. 可观测性生态（Prometheus + Grafana + Loki + Tempo + OpenTelemetry）

这套路线基本覆盖了企业级云原生监控平台从入门到进阶的核心知识，也是当前 Kubernetes 生产环境最主流的监控方案。
