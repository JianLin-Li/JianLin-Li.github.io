八、Pod Security Admission（PSA）

从 Kubernetes v1.25 开始，旧的 PodSecurityPolicy（PSP） 已被移除，官方推荐使用 Pod Security Admission（PSA） 配合命名空间标签来实施 Pod 安全策略。

PSA 提供三种安全级别：

模式 安全等级 说明
Privileged 最宽松 允许特权容器，适合系统组件
Baseline 中等 阻止明显危险配置
Restricted 最严格 推荐普通业务使用，要求非 Root、限制提权等

示例：

1metadata:
2  labels:
3    pod-security.kubernetes.io/enforce: restricted

这样，该命名空间中的 Pod 必须符合 restricted 安全要求，否则创建会被拒绝。

十二、安全最佳实践

分类 推荐做法
Authentication 使用证书或 OIDC，避免匿名访问
Authorization 使用 RBAC，遵循最小权限原则（Least Privilege）
Admission 启用 Pod Security Admission、ResourceQuota、LimitRanger，并按需使用 Webhook
ServiceAccount 每个应用使用独立 ServiceAccount，避免默认账号拥有过高权限
Pod Security Context 禁止 Root、关闭特权模式、禁止提权、启用只读根文件系统、最小化 Capabilities
Secret 管理 使用 Secret 存储敏感信息，避免明文写入镜像或 ConfigMap
NetworkPolicy 默认拒绝（Default Deny），按需开放 Ingress/Egress
镜像安全 使用可信镜像源，定期扫描漏洞，固定镜像版本（Tag/Digest）
审计 开启 API Audit Log，记录关键操作
集群升级 保持 Kubernetes 与 CNI 插件版本更新，及时修复安全漏洞

面试高频总结

问题 核心答案
Kubernetes API 请求要经过哪些阶段？ Authentication → Authorization → Admission Control。
Authentication 和 Authorization 的区别？ Authentication 负责确认身份；Authorization 负责确认权限。
Kubernetes 生产环境常用授权方式？ RBAC。
Admission Controller 的作用？ 在资源写入前对请求进行修改（Mutating）或校验（Validating）。
Pod Security Context 的作用？ 配置 Pod/容器运行时安全属性，如 UID、GID、Capabilities、是否允许提权、是否只读根文件系统等。
PSP 与 PSA 的区别？ PSP 已在 v1.25 移除，官方推荐使用 Pod Security Admission（PSA）。
NetworkPolicy 默认行为是什么？ 未配置策略时通常 Pod 之间默认互通；配置策略后建议采用 Default Deny + 白名单放行。
NetworkPolicy 能控制哪些对象？ 可基于 Pod、Namespace、IPBlock、端口（Port） 对 Ingress/Egress 流量进行控制。

这套内容覆盖了 Kubernetes 安全体系中最核心的知识点，也是 CKA、CKS 认证和企业生产环境中最常考、最常用的安全基础。
