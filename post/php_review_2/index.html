filter_var() 是 PHP 中用于验证（validate）和净化（sanitize）变量的核心函数，属于 PHP Filter 扩展（自 PHP 5.2 起内置，通常默认启用）。它能安全地处理用户输入、URL、邮件、整数等常见数据类型，是防止 XSS、SQL 注入、格式错误等问题的重要工具。

🔹 一、基本语法

1mixed filter_var ( mixed $value [, int $filter = FILTER_DEFAULT [, array $options ]] )
  • $value：要过滤的值（字符串、整数等）
  • $filter：过滤器 ID（常量），决定验证或净化方式
  • $options：可选参数，如标志（flags）、默认值、编码等

✅ 成功时返回过滤后的值；
❌ 失败时返回 false（注意：某些合法值如 0、"0" 也可能被误判，需用 === false 判断）

🔹 二、两大用途：验证（Validate） vs 净化（Sanitize）

类型 目的 行为
验证（Validate） 检查数据是否符合特定格式 符合 → 返回原值；不符合 → false
净化（Sanitize） 清理数据，移除非法字符 总是返回“干净”字符串（可能为空）

🔹 三、常用过滤器常量（Filter Constants）

✅ 1. 通用过滤器

常量 说明
FILTER_DEFAULT 默认，相当于 trim() + 转义（实际等同于 FILTER_UNSAFE_RAW）
FILTER_UNSAFE_RAW 不做任何处理（但可配合 options 转义）
FILTER_SANITIZE_STRING 已废弃（PHP 8.1+），旧版用于移除 HTML/PHP 标签

⚠️ 从 PHP 8.1 起，FILTER_SANITIZE_STRING 被移除！推荐改用 htmlspecialchars() 或 strip_tags()。

✅ 2. 验证类过滤器（Validation）

过滤器 用途 示例
FILTER_VALIDATE_EMAIL 验证邮箱格式 filter_var('a@b.com', FILTER_VALIDATE_EMAIL) → 'a@b.com'
FILTER_VALIDATE_URL 验证 URL 支持 http, https, ftp 等
FILTER_VALIDATE_IP 验证 IP 地址 可指定 IPv4/IPv6
FILTER_VALIDATE_INT 验证整数 可设范围
FILTER_VALIDATE_BOOLEAN 转为布尔值 "true", "1", "on" → true

示例：

 1// 验证邮箱
 2$email = "user@example.com";
 3if (filter_var($email, FILTER_VALIDATE_EMAIL)) {
 4    echo "有效邮箱";
 5}
 6
 7// 验证整数（带范围）
 8$age = "25";
 9$options = ['options' => ['min_range' => 1, 'max_range' => 120]];
10if (filter_var($age, FILTER_VALIDATE_INT, $options) !== false) {
11    echo "年龄合法";
12}

✅ 3. 净化类过滤器（Sanitization）

过滤器 用途 行为
FILTER_SANITIZE_EMAIL 移除邮箱中非法字符 保留 a-z0-9!#$%&'*+-/=?^_{
FILTER_SANITIZE_URL 移除 URL 中非法字符 保留字母、数字和 `$-_.+!*'(),{}
FILTER_SANITIZE_NUMBER_INT 仅保留数字、+、- "abc123def" → "123"
FILTER_SANITIZE_NUMBER_FLOAT 保留浮点数字符 可包含小数点、科学计数法
FILTER_SANITIZE_SPECIAL_CHARS 转义特殊字符（类似 htmlspecialchars） < → &lt;

示例：

1// 净化 URL
2$url = "http://exa<mple>.com/?q=<script>";
3$clean = filter_var($url, FILTER_SANITIZE_URL);
4// 结果: "http://example.com/?q="
5
6// 净化整数
7$input = "123abc456";
8$number = filter_var($input, FILTER_SANITIZE_NUMBER_INT); // "123456"

💡 注意：净化后仍是字符串，如需整数需再转：(int)$number

🔹 四、高级用法：$options 参数

$options 可以是关联数组，常用键：

键 说明
flags 附加标志（如 FILTER_FLAG_IPV4）
options 传递选项（如 min_range）
default 验证失败时返回的默认值（仅部分过滤器支持）
encoding 字符编码（如 "UTF-8"）

示例 1：验证 IPv4

1$ip = "192.168.1.1";
2if (filter_var($ip, FILTER_VALIDATE_IP, FILTER_FLAG_IPV4)) {
3    echo "是 IPv4 地址";
4}

示例 2：设置默认值（PHP 7.0+）

1$value = "invalid";
2$result = filter_var(
3    $value,
4    FILTER_VALIDATE_INT,
5    ["options" => ["default" => 0]]
6);
7// $result = 0

示例 3：组合标志

1// 验证 URL 且要求有查询字符串
2$url = "https://example.com?search=test";
3if (filter_var($url, FILTER_VALIDATE_URL, FILTER_FLAG_QUERY_REQUIRED)) {
4    echo "URL 合法且含查询参数";
5}

🔹 五、常用标志（Flags）

标志 适用过滤器 作用
FILTER_FLAG_IPV4 FILTER_VALIDATE_IP 仅允许 IPv4
FILTER_FLAG_IPV6 FILTER_VALIDATE_IP 仅允许 IPv6
FILTER_FLAG_NO_RES_RANGE FILTER_VALIDATE_IP 禁止私有/保留 IP
FILTER_FLAG_PATH_REQUIRED FILTER_VALIDATE_URL URL 必须包含路径
FILTER_FLAG_QUERY_REQUIRED FILTER_VALIDATE_URL URL 必须包含查询字符串
FILTER_FLAG_STRIP_LOW 净化类 移除 ASCII 值 < 32 的字符
FILTER_FLAG_ENCODE_HIGH 净化类 编码 ASCII > 127 的字符

🔹 六、重要注意事项

1. 不要依赖 filter_var 防 SQL 注入

  • 它不能替代参数化查询（PDO/MySQLi）！
  • 例如：FILTER_SANITIZE_STRING（旧版）不会转义单引号，仍可能导致注入。

2. 验证 vs 净化的选择

  • 验证：用于关键字段（如邮箱、ID），失败应报错。
  • 净化：用于展示内容（如评论、昵称），确保输出安全。

3. false 的陷阱

1var_dump(filter_var("0", FILTER_VALIDATE_INT)); // int(0) → OK
2var_dump(filter_var("", FILTER_VALIDATE_EMAIL)); // bool(false)
3var_dump(filter_var("0", FILTER_VALIDATE_BOOLEAN)); // bool(false)!
4
5// 正确判断：
6if (filter_var($input, FILTER_VALIDATE_EMAIL) === false) {
7    // 无效
8}

4. 邮箱验证不等于可达性

  • FILTER_VALIDATE_EMAIL 只检查格式，不验证邮箱是否存在！

🔹 七、替代方案与现代实践

场景 推荐做法
HTML 输出防 XSS 使用 htmlspecialchars($str, ENT_QUOTES, 'UTF-8')
邮箱验证 filter_var($email, FILTER_VALIDATE_EMAIL) + 发送验证码
整数/浮点数 先 filter_var(..., FILTER_VALIDATE_INT)，再 (int) 转换
复杂数据验证 使用 Symfony Validator、Respect\Validation 等库

✅ 总结：最佳实践清单

  • ✅ 用 FILTER_VALIDATE_* 验证用户输入格式；
  • ✅ 用 FILTER_SANITIZE_* 清理展示内容（注意 PHP 8.1+ 废弃了 SANITIZE_STRING）；
  • ✅ 验证失败时用 === false 判断；
  • ✅ 敏感操作（如数据库）永远使用参数化查询；
  • ✅ 对输出到 HTML 的内容，优先用 htmlspecialchars()；
  • ✅ 不要相信 filter_var 能 100% 防攻击——它只是第一道防线。

在 PHP 中，CURL 本身不支持真正的多线程（因为 PHP 是单线程语言），但可以通过 curl_multi_* 系列函数实现并发（异步）HTTP 请求，从而模拟“多线程”效果，大幅提升多个请求的执行效率（从串行 O(n) 降到接近 O(1)）。

🔹 一、为什么需要 CURL 并发？

  • 串行请求：10 个请求 × 每个 1 秒 = 10 秒
  • 并发请求：10 个请求同时发 → 约 1 秒完成

✅ 适用于：批量 API 调用、爬虫、微服务聚合等场景。

🔹 二、核心函数：curl_multi_*

函数 作用
curl_multi_init() 初始化一个多 handle
curl_multi_add_handle() 添加一个 curl handle 到多 handle
curl_multi_exec() 执行所有请求（非阻塞）
curl_multi_select() 等待 I/O 活动（避免 CPU 空转）
curl_multi_getcontent() 获取某个请求的响应内容
curl_multi_remove_handle() 移除 handle
curl_multi_close() 关闭多 handle

🔹 三、完整示例：并发请求多个 URL

 1<?php
 2
 3function curlMultiRequest($urls, $options = []) {
 4    // 1. 初始化 multi handle
 5    $mh = curl_multi_init();
 6    $handles = [];
 7    $results = [];
 8
 9    // 2. 为每个 URL 创建 curl handle
10    foreach ($urls as $i => $url) {
11        $ch = curl_init();
12        curl_setopt_array($ch, [
13            CURLOPT_URL            => $url,
14            CURLOPT_RETURNTRANSFER => true,
15            CURLOPT_TIMEOUT        => 30,
16            CURLOPT_FOLLOWLOCATION => true,
17            CURLOPT_SSL_VERIFYPEER => false, // 生产环境建议开启
18        ]);
19        
20        // 应用自定义选项
21        if (!empty($options)) {
22            curl_setopt_array($ch, $options);
23        }
24
25        // 添加到 multi handle
26        curl_multi_add_handle($mh, $ch);
27        $handles[$i] = $ch;
28    }
29
30    // 3. 执行并发请求
31    $running = null;
32    do {
33        curl_multi_exec($mh, $running);
34        curl_multi_select($mh); // 阻塞直到有活动（节省 CPU）
35    } while ($running > 0);
36
37    // 4. 收集结果
38    foreach ($handles as $i => $ch) {
39        $content = curl_multi_getcontent($ch);
40        $errno   = curl_errno($ch);
41        $error   = curl_error($ch);
42        $httpCode = curl_getinfo($ch, CURLINFO_HTTP_CODE);
43
44        $results[$i] = [
45            'url'       => $urls[$i],
46            'content'   => $errno ? null : $content,
47            'http_code' => $httpCode,
48            'error'     => $errno ? $error : null,
49        ];
50
51        curl_multi_remove_handle($mh, $ch);
52        curl_close($ch);
53    }
54
55    curl_multi_close($mh);
56    return $results;
57}
58
59// 使用示例
60$urls = [
61    'https://httpbin.org/delay/1',
62    'https://httpbin.org/delay/2',
63    'https://httpbin.org/get',
64];
65
66$start = microtime(true);
67$results = curlMultiRequest($urls);
68$elapsed = microtime(true) - $start;
69
70echo "共 " . count($urls) . " 个请求，并发耗时: " . round($elapsed, 2) . " 秒\n";
71
72foreach ($results as $r) {
73    echo "[{$r['http_code']}] {$r['url']}\n";
74    if ($r['error']) echo "  ❌ Error: {$r['error']}\n";
75}

💡 输出示例：

1共 3 个请求，并发耗时: 2.15 秒
2[200] https://httpbin.org/delay/1
3[200] https://httpbin.org/delay/2
4[200] https://httpbin.org/get

⚠️ 如果串行执行，至少需要 1+2+0.5 ≈ 3.5 秒。

🔹 四、高级技巧

✅ 1. 限制并发数量（防资源耗尽）

当 URL 数量很大（如 1000+），可分批处理：

 1function curlMultiBatch($urls, $concurrency = 10, $options = []) {
 2    $results = [];
 3    $total = count($urls);
 4    
 5    for ($i = 0; $i < $total; $i += $concurrency) {
 6        $batch = array_slice($urls, $i, $concurrency);
 7        $batchResults = curlMultiRequest($batch, $options);
 8        $results = array_merge($results, $batchResults);
 9    }
10    
11    return $results;
12}

✅ 2. 设置超时和重试

在 $options 中添加：

1$options = [
2    CURLOPT_TIMEOUT        => 10,
3    CURLOPT_CONNECTTIMEOUT => 5,
4    // 可配合自定义重试逻辑
5];

✅ 3. 获取详细信息（headers、redirect 等）

1curl_setopt($ch, CURLOPT_HEADER, true);
2// 或
3$info = curl_getinfo($ch);

在 PHP 的 cURL 请求中，SSL 配置和 HTTP 认证（Basic/Digest） 是两个常见且重要的安全设置。下面分别详解如何正确配置，并提供完整示例。

🔐 一、cURL 设置 SSL（HTTPS）

当请求 https:// 网站时，PHP cURL 默认会验证 SSL 证书。若配置不当，可能报错：

cURL error: SSL certificate problem: unable to get local issuer certificate

✅ 正确做法（生产环境推荐）

方法 1：使用系统 CA 证书包（推荐）

1$ch = curl_init();
2curl_setopt_array($ch, [
3    CURLOPT_URL            => 'https://example.com',
4    CURLOPT_RETURNTRANSFER => true,
5    CURLOPT_SSL_VERIFYPEER => true,   // 验证证书是否由可信 CA 签发
6    CURLOPT_SSL_VERIFYHOST => 2,      // 验证证书中的域名是否匹配
7    CURLOPT_CAINFO         => '/etc/ssl/certs/ca-certificates.crt', // Linux 常见路径
8    // Windows 可用: 'C:\php\extras\ssl\cacert.pem'
9]);

📌 CA 证书文件获取：

  • 从 curl.se/docs/caextract.html 下载 cacert.pem
  • 放到服务器（如 /usr/local/etc/openssl/cert.pem 或自定义路径）
  • 在 php.ini 中全局设置（可选）： > ```ini curl.cainfo = "/path/to/cacert.pem" openssl.cafile = "/path/to/cacert.pem"

方法 2：禁用 SSL 验证（仅限开发/测试！）

1curl_setopt_array($ch, [
2    CURLOPT_SSL_VERIFYPEER => false,
3    CURLOPT_SSL_VERIFYHOST => false,
4]);

⚠️ 绝对不要在生产环境使用！ 会导致中间人攻击（MITM）风险。

🔑 二、cURL 设置 HTTP 认证

HTTP 认证常见于 API、管理后台等需要用户名/密码的场景。

1. HTTP Basic Auth

最简单，Base64 编码 username:password 放入 Authorization 头。

1$ch = curl_init();
2curl_setopt_array($ch, [
3    CURLOPT_URL            => 'https://api.example.com/data',
4    CURLOPT_RETURNTRANSFER => true,
5    CURLOPT_USERPWD        => 'myuser:mypassword', // 格式: "username:password"
6    CURLOPT_HTTPAUTH       => CURLAUTH_BASIC,
7]);

💡 等价于手动设置 Header：

1curl_setopt($ch, CURLOPT_HTTPHEADER, [
2    'Authorization: Basic ' . base64_encode('myuser:mypassword')
3]);

2. HTTP Digest Auth

比 Basic 更安全（避免明文传输密码），服务器会挑战（challenge）客户端。

1curl_setopt_array($ch, [
2    CURLOPT_URL            => 'https://secure.example.com/api',
3    CURLOPT_RETURNTRANSFER => true,
4    CURLOPT_USERPWD        => 'alice:secret123',
5    CURLOPT_HTTPAUTH       => CURLAUTH_DIGEST,
6]);

✅ cURL 会自动处理 WWW-Authenticate 挑战流程。

3. 支持多种认证方式（自动协商）

1curl_setopt($ch, CURLOPT_HTTPAUTH, CURLAUTH_BASIC | CURLAUTH_DIGEST);

自签名 SSL 证书完整指南（含多域名与通配符支持）

创建自签名 SSL 证书（Self-Signed Certificate）是开发、测试或内部服务中常见的需求。虽然它不被浏览器或操作系统信任（会显示“不安全”警告），但在本地开发、内网服务、Docker 容器通信等场景非常实用。

下面以 OpenSSL（几乎所有 Linux/macOS 系统自带，Windows 可通过 WSL、Git Bash 或安装 OpenSSL）为例，详细介绍创建过程。

🔐 一、基本概念

  • 私钥（Private Key）：server.key —— 必须保密，用于解密和签名。
  • 证书签名请求（CSR）：server.csr —— 可选，用于向 CA 申请证书（自签名时可跳过）。
  • 自签名证书（Certificate）：server.crt —— 包含公钥和身份信息，由自己签发。

✅ 自签名证书 = 私钥 + 自己给自己签名的公钥证书。

🛠 二、快速创建（一步生成）

命令（推荐用于开发）：

1openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
2  -keyout server.key -out server.crt \
3  -subj "/C=CN/ST=Beijing/L=Beijing/O=MyOrg/CN=localhost"

参数说明：

参数 作用
-x509 直接生成自签名证书（跳过 CSR）
-nodes 不要加密私钥（否则每次用都要输密码）
-days 365 证书有效期 365 天
-newkey rsa:2048 生成 2048 位 RSA 私钥
-keyout server.key 输出私钥文件
-out server.crt 输出证书文件
-subj "..." 自动填写证书信息（避免交互式输入）

💡 常见字段含义：

  • C = Country（国家，如 CN）
  • ST = State（省）
  • L = Locality（市）
  • O = Organization（组织）
  • CN = Common Name（必须匹配你访问的域名或 IP，如 localhost、192.168.1.100、myapp.local）

🧪 三、详细步骤（含 CSR，更规范）

步骤 1：生成私钥

1openssl genrsa -out server.key 2048

步骤 2：生成 CSR（证书签名请求）

1openssl req -new -key server.key -out server.csr \
2  -subj "/C=CN/ST=Beijing/L=Beijing/O=MyOrg/CN=localhost"

步骤 3：用私钥自签名 CSR，生成证书

1openssl x509 -req -days 365 -in server.csr -signkey server.key -out server.crt

✅ 这种方式适合需要 CSR 的场景（如未来要替换为正式 CA 证书）。

🌐 四、支持多个域名 / IP（SAN 扩展）

现代浏览器要求证书包含 Subject Alternative Name (SAN)，否则即使 CN=localhost 也会报错。

方法：使用配置文件

  1. 创建 server.conf：
 1[req]
 2default_bits = 2048
 3prompt = no
 4default_md = sha256
 5distinguished_name = dn
 6req_extensions = v3_req
 7
 8[dn]
 9C = CN
10ST = Beijing
11L = Beijing
12O = MyOrg
13CN = localhost
14
15[v3_req]
16basicConstraints = CA:FALSE
17keyUsage = nonRepudiation, digitalSignature, keyEncipherment
18subjectAltName = @alt_names
19
20[alt_names]
21DNS.1 = localhost
22DNS.2 = myapp.local
23IP.1 = 127.0.0.1
24IP.2 = 192.168.1.100
  1. 生成证书：
1openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
2  -keyout server.key -out server.crt \
3  -config server.conf -extensions v3_req

✅ 这样生成的证书支持 https://localhost、https://myapp.local、https://127.0.0.1 等。

🖥 五、在 Web 服务器中使用

Nginx 示例：

 1server {
 2    listen 443 ssl;
 3    server_name localhost;
 4
 5    ssl_certificate /path/to/server.crt;

 8    location / {
 9        root /var/www/html;
10    }
11}

🔒 六、让系统/浏览器信任自签名证书（可选）

⚠️ 仅建议用于开发机或内网环境！

macOS：

  2. 在“钥匙串访问”中找到该证书 → 右键 → “显示简介” → 展开“信任” → 选择“始终信任”

2sudo update-ca-certificates

💡 浏览器可能仍需重启才能生效。

查看证书内容：

验证 HTTPS 服务：

✅ 总结

场景 推荐命令
快速开发 一步生成（带 -subj）
多域名/IP 使用 server.conf 配置 SAN
生产环境 不要用自签名！ 请用 Let's Encrypt（免费）或商业 CA

🔑 关键提醒：

  • 自签名证书不能用于公网生产环境（用户会看到严重警告）
  • 开发时可用，但务必配合 CN 或 SAN 匹配访问地址
  • 私钥（.key）切勿泄露！

支持多个域名的自签名证书（如 php.demo.com 和 php.myself.com）

要让一个自签名 SSL 证书同时支持多个域名（如 php.demo.com 和 php.myself.com），必须使用 SAN（Subject Alternative Name，主题备用名称）扩展。现代浏览器和客户端（包括 curl、PHP stream、Node.js 等）都要求使用 SAN，仅靠 Common Name (CN) 已不再被信任。

下面以你的两个域名为例：php.demo.com 和 php.myself.com，详细说明如何生成支持多域名的自签名证书。

✅ 步骤 1：创建 OpenSSL 配置文件（关键！）

创建一个配置文件，比如叫 php-sites.conf：

 1[req]
 2default_bits       = 2048
 3prompt             = no
 4default_md         = sha256
 5distinguished_name = dn
 6req_extensions     = v3_req
 7
 8[dn]
 9C  = CN
10ST = Guangdong
11L  = Shenzhen
12O  = MyDevTeam
13CN = php.demo.com
14
15[v3_req]
16basicConstraints   = CA:FALSE
17keyUsage           = nonRepudiation, digitalSignature, keyEncipherment
18subjectAltName     = @alt_names
19
20[alt_names]
21DNS.1 = php.demo.com
22DNS.2 = php.myself.com

🔍 说明：

  • [dn] 中的 CN（Common Name）可以任选其一（或写一个通用名），但实际生效的是 [alt_names]
  • DNS.1, DNS.2... 列出所有你要支持的域名
  • 如果还需要支持 IP 地址，可加 IP.1 = 192.168.1.100（但你这里不需要）

✅ 步骤 2：使用 OpenSSL 生成证书

在终端执行以下命令（确保 php-sites.conf 在当前目录）：

1openssl req -x509 -nodes -days 365 \
2  -newkey rsa:2048 \
3  -keyout php-sites.key \
4  -out php-sites.crt \
5  -config php-sites.conf \
6  -extensions v3_req

参数解释：

参数 作用
-x509 直接生成自签名证书
-nodes 私钥不加密（避免每次启动服务输密码）
-days 365 有效期 1 年
-config php-sites.conf 使用自定义配置
-extensions v3_req 应用配置中的 v3_req 扩展（包含 SAN）

执行后会生成两个文件：

  • php-sites.key → 私钥（保密！）
  • php-sites.crt → 自签名证书（可分发）

✅ 步骤 3：验证证书是否包含两个域名

运行以下命令查看证书详情：

1openssl x509 -in php-sites.crt -text -noout

在输出中找到 X509v3 Subject Alternative Name 部分，应看到：

1X509v3 Subject Alternative Name:
2    DNS:php.demo.com, DNS:php.myself.com

✅ 如果看到这两个域名，说明配置成功！

✅ 步骤 4：在 Web 服务器中使用（以 Nginx 为例）

 1server {
 2    listen 443 ssl http2;
 3    server_name php.demo.com php.myself.com;
 4
 5    ssl_certificate     /path/to/php-sites.crt;
 6    ssl_certificate_key /path/to/php-sites.key;
 7
 8    root /var/www/php-demo;
 9    index index.php;
10
11    location ~ \.php$ {
12        fastcgi_pass unix:/run/php/php8.2-fpm.sock;
13        include fastcgi_params;
14        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
15    }
16}

💡 注意：

  • server_name 要包含两个域名
  • 证书和私钥路径要正确

然后重载 Nginx：

1sudo nginx -t && sudo systemctl reload nginx

✅ 步骤 5：本地测试（修改 hosts）

因为 php.demo.com 和 php.myself.com 不是公网域名，需在本机 hosts 文件中映射到服务器 IP（如 127.0.0.1）：

Linux/macOS：

1sudo nano /etc/hosts

添加：

1127.0.0.1 php.demo.com
2127.0.0.1 php.myself.com

Windows：

编辑 C:\Windows\System32\drivers\etc\hosts，加上同样两行。

保存后，浏览器访问：

  • https://php.demo.com
  • https://php.myself.com

⚠️ 第一次访问仍会提示“不安全”（因为是自签名），点击“高级” → “继续前往”即可。

如果你希望彻底消除警告，可将 php-sites.crt 安装为受信任的根证书（见下文）。

🔒 （可选）让系统信任该证书

macOS：

  1. 双击 php-sites.crt → 添加到“系统”钥匙串
  2. 打开“钥匙串访问” → 找到该证书 → 右键 → “显示简介”

  2. 选择“本地计算机” → 存入“受信任的根证书颁发机构”

1sudo cp php-sites.crt /usr/local/share/ca-certificates/php-sites.crt
2sudo update-ca-certificates

🔄 安装后重启浏览器，访问两个域名将不再有安全警告！

❌ 常见错误排查

问题 原因 解决
浏览器报 ERR_CERT_COMMON_NAME_INVALID 证书没有 SAN 或 SAN 未包含该域名 检查 openssl x509 -text 输出
curl 报 SSL: no alternative certificate subject name matches 同上 重新生成带正确 SAN 的证书

✅ 总结

要支持多个域名，核心就是配置 subjectAltName：

  1. 写一个 .conf 文件，列出所有 DNS.x = your.domain
  2. 用 openssl req -x509 -config your.conf -extensions v3_req 生成
  3. 验证 SAN 是否生效
  4. 配置 Web 服务器 + hosts 测试

🎯 最终效果：
一个证书 → 同时安全地服务于 https://php.demo.com 和 https://php.myself.com

生成支持通配符域名（Wildcard Domain）的自签名证书

要生成支持通配符域名（Wildcard Domain） 的 SSL 证书（例如 *.demo.com），你仍然可以使用 OpenSSL 自签名证书，但需注意以下关键点：

✅ 一、通配符证书的规则

  • 通配符 只能匹配一级子域名：
    • *.demo.com ✅ 匹配：a.demo.com、api.demo.com、php.demo.com
    • *.demo.com ❌ 不匹配：demo.com（主域名）、a.b.demo.com（二级子域）
  • 通配符 必须出现在最左侧标签：* 只能代替一个完整的 DNS 标签（不能写成 api.*.com）
  • 在 SAN（Subject Alternative Name）中指定通配符才有效（仅 CN=*.demo.com 已被现代浏览器忽略）

✅ 二、生成支持通配符的自签名证书（以 *.demo.com 为例）

步骤 1：创建 OpenSSL 配置文件 wildcard.conf

 1[req]
 2default_bits       = 2048
 3prompt             = no
 4default_md         = sha256
 5distinguished_name = dn
 6req_extensions     = v3_req
 7
 8[dn]
 9C  = CN
10ST = Guangdong
11L  = Shenzhen
12O  = MyDevTeam
13CN = *.demo.com
14
15[v3_req]
16basicConstraints   = CA:FALSE
17keyUsage           = nonRepudiation, digitalSignature, keyEncipherment
18subjectAltName     = @alt_names
19
20[alt_names]
21DNS.1 = *.demo.com
22DNS.2 = demo.com          # 👈 如果你也想支持主域名，需显式添加！

🔍 说明：

  • DNS.1 = *.demo.com → 支持所有一级子域
  • DNS.2 = demo.com → 单独添加主域名（通配符不包含它！）
  • 如果你还有其他固定子域（如 admin.demo.com），也可以加，但非必需

步骤 2：生成证书和私钥

1openssl req -x509 -nodes -days 365 \
2  -newkey rsa:2048 \
3  -keyout wildcard-demo.key \
4  -out wildcard-demo.crt \
5  -config wildcard.conf \
6  -extensions v3_req

执行后生成：

  • wildcard-demo.key：私钥
  • wildcard-demo.crt：自签名证书

步骤 3：验证证书是否包含通配符 SAN

1openssl x509 -in wildcard-demo.crt -text -noout | grep -A1 "Subject Alternative Name"

✅ 正确输出应为：

1X509v3 Subject Alternative Name:
2    DNS:*.demo.com, DNS:demo.com

⚠️ 如果只看到 CN = *.demo.com 但没有 SAN，浏览器会拒绝！

✅ 三、在 Web 服务器中使用（Nginx 示例）

 1server {
 2    listen 443 ssl http2;
 3    server_name ~^.*\.demo\.com$ demo.com;  # 匹配所有子域 + 主域
 4
 5    ssl_certificate     /path/to/wildcard-demo.crt;
 6    ssl_certificate_key /path/to/wildcard-demo.key;
 7
 8    root /var/www/demo;
 9    index index.php;
10
11    location ~ \.php$ {
12        fastcgi_pass unix:/run/php/php8.2-fpm.sock;
13        include fastcgi_params;
14        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
15    }
16}

💡 server_name 使用正则 ~^.*\.demo\.com$ 或直接列出常用子域：

1server_name demo.com *.demo.com;

（Nginx 从 1.15.7+ 开始支持 *.example.com 语法）

✅ 四、本地测试（修改 hosts）

因为通配符无法在 hosts 文件中直接模拟，你需要为每个测试子域单独添加记录：

1# /etc/hosts
2127.0.0.1 demo.com
3127.0.0.1 php.demo.com
4127.0.0.1 api.demo.com
5127.0.0.1 test.demo.com

然后访问：

  • https://php.demo.com ✅
  • https://api.demo.com ✅
  • https://demo.com ✅（因为我们显式添加了）
  • https://a.b.demo.com ❌（通配符不支持多级）

⚠️ 五、重要限制与注意事项

项目 说明
不能用于公网生产环境 自签名证书会被浏览器标记为不安全
Let's Encrypt 支持免费通配符证书 但需要 DNS API 验证（如 Cloudflare、阿里云）
通配符不包含主域名 必须在 SAN 中显式添加 demo.com
不支持多级通配符 *.*.demo.com 无效
私钥安全 通配符私钥一旦泄露，所有子域都危险！

🌐 六、生产环境建议：使用 Let's Encrypt（免费）

如果你有公网域名和 DNS 控制权，推荐用 Certbot + DNS 插件申请免费通配符证书：

1# 示例：使用 Cloudflare DNS
2sudo certbot certonly \
3  --dns-cloudflare \
4  --dns-cloudflare-credentials ~/.secrets/cloudflare.ini \
5  -d demo.com -d '*.demo.com'

✅ 优点：被所有浏览器信任，自动续期。

✅ 总结

目标 操作
支持 *.demo.com 在 SAN 中添加 DNS.1 = *.demo.com
同时支持 demo.com 额外添加 DNS.2 = demo.com
本地开发测试 修改 hosts + 自签名证书 + 安装为受信任根证书
公网生产环境 使用 Let's Encrypt 申请免费通配符证书

🔑 核心口诀：
“通配符只管一级子域，主域要单独加，SAN 是必须项！”
